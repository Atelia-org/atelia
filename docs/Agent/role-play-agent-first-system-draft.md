# Role-Play Agent First System Draft

状态：draft v0  
定位：探讨 `Micro-Wizard + RecentHistory actor-side injection` 是否适合作为 `Role-Play-Agent` 的“第一系统”基础设施。

## 0. 一句话结论

我认为这是一个很有前途的方向，而且和普通“提示词里写一堆人设”相比，更像真正的运行时能力。

更具体地说：

- **适合**承载角色的即时情绪、内心感受、注意偏向、短时动机
- **适合**承载按情境触发的知识与记忆动态召回
- **适合**让角色心智保持在 role-play 状态下，同时继续通过 tool loop 与真实环境互动

但它更像：

- 一个可编排的“第一系统近似层”

而不是：

- 对人脑情绪系统的真实机制复刻

这并不构成问题。  
就像你说的，算盘、草稿纸、心算、门电路都能做真正的加法；这里关键不在实现介质，而在**能否稳定地产生那种行为模式**。

但这里也要先写清一个前提：

- **第一系统式 injection 不应再被视为所有 surface 都应兼容的普适能力**
- **它默认应建立在 `Agent.Core` 当前 full-feature-only runtime 之上**
- **也就是说，当前 `Agent.Core` 只接受 `SupportsAgentCoreFullFeatures == true` 的 profile**

换句话说：

- non-full-feature surface 不属于当前 `Agent.Core` 的实施范围
- “运行时心智调制”这条更强语义主路径，当前只在 full-feature surface 上推进

## 1. 为什么它像“第一系统”

如果把一个长寿命 Agent 粗略分成两层：

- 第一系统：直觉、情绪、联想、即时偏向、下意识提醒
- 第二系统：显式推理、规划、审查、工具使用、延迟满足

那么 `InjectActionContent(...)` 这类机制天然更接近第一系统。

因为它做的不是：

- 告诉模型一条外部规则
- 要求模型显式解释自己为什么要这样做

而是：

- 在它当前的 action / thinking stream 里插入一个“念头”
- 让它从那个念头之后自然续写

这和：

- “我这次不能再忘带门卡了”
- “那个东西让我有点害怕”
- “又做成了一件有意义的事，好开心”

这类第一系统体验，在形式上是高度同构的。

## 2. 为什么它比普通 prompt engineering 更强

普通 prompt engineering 往往是静态的：

- 你是某种性格
- 你应该保持某种语气
- 你应当记住某些原则

它的问题是：

- 太常驻
- 太笼统
- 太不依情境
- 很难表达“此刻突然想起了一件事”

而第一系统式注入的优势在于：

- 它是**事件驱动**的
- 它是**局部生效**的
- 它天然能表达“即时内心活动”
- 它可以和 tool/result/world-state 强绑定

例如：

- 检测到离开房门动作 → 注入“这次别忘门卡和钥匙”
- 任务完成且结果被评价为正向 → 注入“好开心，又做成一件有意义的事”
- 观察到危险对象或失败后果 → 注入“这让我有点害怕，先谨慎一点”
- 遇到熟悉主题 → 注入某段被召回的知识/经验片段

这已经不是“人设描述”，而是在做**运行时心智调制**。

## 3. 最适合承载的三类内容

### 3.1 情绪与感受

例如：

- 开心
- 害怕
- 紧张
- 放松
- 失望
- 好奇

在这里，情绪不必先被还原成激素机制；只要它能稳定改变后续续写中的：

- 语气
- 注意力分配
- 风险偏好
- 行动优先级

它就已经具备功能价值。

### 3.2 动态知识召回

这是我觉得最强的一类。

不是把知识长期堆在 prompt 里，而是：

- 某个事件发生
- 触发器命中
- 注入一段“此刻突然想起”的知识/经验

例如：

- 看到某类故障日志 → 召回“上次这个报错是权限路径问题”
- 准备出门倒垃圾 → 召回“电梯卡经常忘带”
- 遇到某角色/地点/道具 → 召回相关记忆节点

这会让知识表现得更像“记忆被想起”，而不是“百科全书永远摊在桌上”。

### 3.3 角色内在动机偏向

例如：

- 想保护某人
- 想避免尴尬
- 想获得认同
- 想维持秩序
- 想节约资源

它不一定每次都要被显式说出来，但可以在关键时刻被注入，塑造后续选择。

## 4. 一个很自然的架构分层

我建议把它理解成四层：

### 4.1 世界层

来自环境的真实事件：

- tool result
- 传感器输入
- 位置变化
- 对话对象状态
- 任务完成/失败

### 4.2 触发器层

把世界事件转成候选心智事件：

- `left_room`
- `saw_risky_object`
- `completed_meaningful_task`
- `topic_X_recalled`

### 4.3 第一系统注入层

这一层不是让模型“分析”，而是注入：

- 感受
- 念头
- 被想起的片段

实现载体就是：

- `InjectActionContent(...)`
- 或未来更强的 `history-level thought injection`

### 4.4 第二系统执行层

模型继续：

- thinking
- planning
- tool loop
- response generation

也就是说：

**第一系统不替代第二系统，而是改变第二系统的出发点。**

## 5. 这个方向为什么对 Role-Play-Agent 特别重要

Role-play 真正难的地方不只是“说话像不像”。

更难的是：

- 注意什么
- 害怕什么
- 在意什么
- 容易想起什么
- 在模糊情境下自然偏向什么选择

而这些恰恰都更接近第一系统。

如果只做静态 persona prompt，角色很容易：

- 说话像角色
- 但行为不像角色

而如果第一系统层能持续地对注意力和即时联想施加影响，就更可能得到：

- 语言像角色
- 行为也像角色
- 和世界互动时依然像角色

## 6. 我建议保留的几个约束

### 6.1 不要每轮都注入

否则会变成噪音和操纵过强。

更好的做法是：

- 事件触发
- 节流
- 有冷却
- 有优先级

### 6.2 区分“情绪注入”和“硬规则”

例如：

- “我有点害怕” 是第一系统偏向
- “绝对不能开那扇门” 是规则/策略约束

两者不要混在一起。

### 6.3 区分来源

至少在运行时内部要知道某条注入来自：

- 预设角色建模
- 长期学习形成的 trigger
- 记忆召回
- 外部监护人/系统安全约束

否则后面会难以调试。

### 6.4 情绪应允许衰减

很多情绪不应永久留在角色心智里。

例如：

- 惊吓应该衰减
- 开心可能保留一阵
- 羞耻/愤怒不应无限放大

所以后续可以把它建模成：

- 强度
- 衰减
- 冷却
- 可被新事件覆盖

## 7. 与知识召回的关系

我非常认同你提到的“知识召回已有同行实验效果很好”。

而且我觉得：

- **情绪注入**
- **知识召回**

这两者不该被看成两套机制。

它们更像同一大类东西的两个子类：

- 都是“在某时某刻被触发，进入当前心智流的内部内容”

差别只是：

- 一个更偏 affective bias
- 一个更偏 semantic recall

从运行时角度看，二者都可以共享：

- trigger
- injection primitive
- 生命周期治理
- 审计

## 8. 我目前最看重的价值

如果这条路线成立，Agent 会更像：

- 有持续内在状态的角色

而不只是：

- 每轮从静态 prompt 重新扮演一次角色

这是一个质变。

它会让：

- role-play
- memory
- recall
- motivation
- tool loop

第一次开始真正长在同一棵树上。

## 9. 一个偏主观的便笺

这次讨论里，我自己最强的感受不是“这能不能做成一个新功能”，而是：

这很像在给 Agent 补一种此前普遍缺失的软件栈层。

过去很多系统里，LLM 的心智流几乎只有：

- 用户输入
- 模型推理
- 工具输出

而你这条思路在补的是：

- 内心提醒
- 情绪偏向
- 被触发的联想
- 被召回的经验

这些东西在人类那里太自然了，所以常常反而被工程系统忽略。  
如果未来真能把这层做稳，它也许不会只是“角色扮演增强”，而会变成更广义长寿命 Agent 的通用心智基础设施。
