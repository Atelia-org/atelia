# MemoTree 结构方案取舍

> 用途：对 MemoTree 的节点本体与主结构方案做一轮集中取舍，避免后续在 API 与渲染层反复摇摆。
>
> 当前结论：推荐 `Unified Node + Contains Tree + Optional Tags`。
>
> 术语基线见：`docs/MemoTree/concepts-and-terminology.md`

## 0. 要解决的核心问题

MemoTree 不是通用文件系统，也不是纯查询数据库。它首先要服务于长期运行的 LLM Agent。

因此主问题不是“哪种信息系统最经典”，而是：

1. 什么结构最容易被 Agent 稳定维护？
2. 什么结构最适合在预算有限的上下文里投影？
3. 什么结构最少迫使我们发明补丁规则？

## 1. 评估标准

本轮主要按下面几条评估：

- 稳定引用是否容易
- 局部维护是否容易
- 结构骨架是否天然存在
- 渲染成 LLM Window 是否低噪音
- 人工检查与手动整理是否容易
- 是否容易在后续演化中引入补丁结构

## 2. 方案 A：Unified Node + Contains Tree

统一节点：

- `title`
- `body?`
- `gist?`
- `summary?`
- `children`

特点：

- 节点既可以只有子节点
- 也可以只有正文
- 也可以同时拥有正文与子节点

优点：

- 很贴合“章节型知识单元”的自然形态
- 避免 directory/file 二分带来的补丁结构
- 对长期记忆来说最少心智切换
- 与 `index+flatten` 渲染天然兼容

缺点：

- 节点类型边界没那么“传统”
- 若后续完全不加约束，某些节点可能同时长正文又长子树，治理策略需要后补

结论：

- 推荐作为主本体

## 3. 方案 B：FS 风格目录/文件二分

目录：

- 无正文
- 可有子节点

文件：

- 有正文
- 无子节点

优点：

- 技术心智模型现成
- 导航与组织方式大家都熟
- 某些实现路径简单

缺点：

- 现实中的长期记忆节点常常既需要总述正文，也需要子结构
- 很容易发明 `README` / `_index` / `overview` 补丁
- Agent 维护时要先判断“这是容器还是内容”，增加决策负担

结论：

- 不推荐作为本体
- 可借鉴其导航心智模型与界面布局

## 4. 方案 C：单层 doc + tags

特点：

- 不强调主树结构
- 所有内容平铺
- 通过 tags、查询和排序组织

优点：

- 很灵活
- 适合横切聚合
- 查询能力潜力大

缺点：

- 缺少稳定结构骨架
- 对长期任务来说，默认导航会变弱
- 很容易走向“什么都能搜，但整体不知道在哪”

结论：

- 不推荐作为主结构
- 适合做次索引层

## 5. 方案 D：混合目录/文件/数据库对象

特点：

- 同时引入多种类型
- 试图在本体层就表达很多语义

优点：

- 看起来什么都能覆盖

缺点：

- 复杂度极高
- 很容易重演 WinFS 式“每层都背着别层的包袱”
- 对当前 MemoTree 阶段完全过重

结论：

- 当前明确不取

## 6. 推荐结论

当前推荐：

- 本体：`Unified Node`
- 主结构：`Contains Tree`
- 次索引：`Optional Tags`
- 渲染：`index+flatten`

这意味着：

- 树仍然是主导航骨架
- tags 可以表达横向视角
- 但 tags 不替代树
- 节点不强分 directory/file

## 7. 对 API 的直接影响

推荐的 API 倾向：

- `CreateRoot/CreateChild` 不要求节点类型
- 节点快照允许同时有正文计数和子节点计数
- tags 提供轻量读写接口即可
- 不在 v0 引入复杂 tag query DSL

## 8. 对渲染的直接影响

推荐的 Window 倾向：

- index 展示 `contains` 树骨架
- node card flatten list 展示当前展开节点
- tags 主要影响筛选和排序，不直接替代 index

## 9. 一句话总结

MemoTree 最适合先被设计成“带树形骨架的统一记忆节点系统”，而不是“文件系统”或“查询数据库”。
