# 山谷小村场景草案

> 用途：把“山谷小村”从推荐场景推进成可用于反推数据模型、事件语义与 user story 的具体工作面。

## 一句话定义

山谷小村是一个 20 到 40 人规模的半封闭聚落：

- 生计依赖农田、林地、水源和少量手工业
- 对外连通主要通过一个山口
- 村内事务高度重复，但永远不完全静止
- 绝大多数关键行为可在地点图层面表达

它的价值不在于“剧情感”，而在于它能稳定地产生：

- 协作
- 分配
- 迟到与错过
- 承诺与违约
- 观察差异
- 谣言与误判
- 追逐与回避

这些正好是当前 SimWorld 最值得优先锻炼的能力。

> 注：本文暂时沿用“村民”“住户”等人类社会友好的称呼，但这不构成“主体必须是生物学人类”的设计承诺。具体见 [inhabitant-profiles.md](inhabitant-profiles.md)。

## 设计主张

### 1. 这不是田园风背景板，而是一个会持续冒出协调问题的社会系统

如果山谷小村只是“平静村庄”，它很快会退化成：

- 排班器
- 对话生成器
- 温和的剧情容器

它要成立，必须持续存在张力来源：

- 资源不够充裕
- 劳力并不稳定
- 信息并不同时到达所有人
- 事情往往来不及等所有人商量完再做

### 2. 核心复杂度应落在“社会协调”而不是“房间几何”

这个场景的第一阶段不依赖：

- 房间内格子
- 家具级遮挡
- 复杂物理碰撞
- 逐步朝向控制

而应优先依赖：

- 谁现在在哪
- 哪两处之间能否通行/看见
- 谁持有关键工具或资源
- 谁接到了什么任务
- 谁知道什么、误以为什么

### 3. 小世界必须有“瓶颈”和“回路”

如果地点图只有单一路径，很多行为会过于平板。

山谷小村最好同时具备：

- 一个明显的对外瓶颈：山口
- 一个高频公共节点：中央空地
- 一条不那么公开的回路：后坡小路

这样才能在不引入地点内几何的前提下，产生：

- 守门
- 绕行
- 巡逻
- 错过
- 尾随
- 逃离

## 形状草图

### 村庄轮廓

- 地形上是一个相对封闭的小山谷
- 对外出入口主要是北侧山口
- 村内有一片集中居住区和若干公共功能点
- 农田与林地分别位于村外缘的不同方向
- 存在一条不经过中央空地的后坡小路，作为低可见度绕行路径

### 推荐的 MVP 地点图

首版建议使用 10 个地点：

| 地点 | 作用 | 备注 |
|------|------|------|
| 山口哨点 | 对外出入口、观察来客 | 高瓶颈、高可见性 |
| 村口主路 | 山口与村内的过渡节点 | 高通行、高可见性 |
| 中央空地 | 公共汇合、分配、通报 | 村内最公开节点 |
| 住屋区 | 日常休息、家庭活动 | 中等可见性 |
| 水井 | 取水、偶遇、消息交换 | 高频短暂停留 |
| 谷仓 | 粮食存放、争议高发点 | 资源敏感地点 |
| 工坊 | 修理、借还工具 | 工具与任务耦合点 |
| 农田 | 稳定劳动地点 | 白天高活跃 |
| 林地入口 | 采集、砍柴、边缘活动 | 半公开边界 |
| 后坡小路 | 绕行、尾随、回避 | 低可见性，较慢 |

### 推荐连接关系

建议至少有以下边：

- 山口哨点 <-> 村口主路
- 村口主路 <-> 中央空地
- 中央空地 <-> 住屋区
- 中央空地 <-> 水井
- 中央空地 <-> 谷仓
- 中央空地 <-> 工坊
- 中央空地 <-> 农田
- 农田 <-> 林地入口
- 林地入口 <-> 后坡小路
- 后坡小路 <-> 住屋区

这张图故意不是树，而是一个带回路的地点网。

## 边语义建议

地点图的边不应该只是“能不能走”，还应携带少量高价值语义。

MVP 先保留三类边就够用：

| 边类型 | 典型例子 | 语义 |
|------|------|------|
| 公开边 | 村口主路、中央空地到水井 | 高可见性，行动容易被观察 |
| 常规边 | 中央空地到住屋区、工坊、谷仓 | 默认通行，观察面一般 |
| 隐蔽边 | 后坡小路 | 可绕行，较低可见性，耗时更长 |

MVP 的“遮挡/阻挡”主要通过以下方式表现：

- 边不可通行
- 边不可视通
- 边耗时变长
- 边需要条件才能通过

这样即便没有地点内格子，也足以表达：

- 山口封锁
- 夜间巡查
- 绕过公共区域
- 躲开主要视线

## 世界中的持续压力

山谷小村之所以好用，在于它天然有几个持续运转的压力源。

### 1. 食物压力

- 农田产出并不总是稳定
- 谷仓库存会下降
- 分配决策会影响谁去做别的事

### 2. 劳力压力

- 同一时段可用的人手有限
- 生病、受伤、迟到、失踪会直接扰动任务安排

### 3. 工具压力

- 好工具稀缺
- 借还关系天然引出冲突、责任和补位

### 4. 时间压力

- 天黑前必须做完某些事
- 某人出发太晚就会错过另一个人
- 同一任务若被推迟，会连锁影响后面事件

### 5. 信息压力

- 不是每个人都同时知道发生了什么
- 谣言经常快于核实
- 同一事件会产生不同版本的观察

这五种压力叠在一起，已经足以支撑长期涌现。

## 资源循环

首版不需要完整经济系统，但需要至少一条能稳定运转的资源循环。

### MVP 资源集

- 食物
- 水
- 木柴
- 工具
- 劳力时间

### 基本循环

1. 村民消耗食物和水
2. 农田提供基础食物补给
3. 林地入口关联木柴与简易材料
4. 工坊维修和恢复工具状态
5. 工具与劳力共同决定任务能否执行

其中最关键的不是“产量模拟得多真实”，而是：

- 某个资源短缺会真实改变行为选择
- 某个工具被占用会真实影响事件结果
- 某个劳动力缺席会真实触发重排

## 角色生态

这个世界不需要一开始就有复杂职业系统，但需要一组稳定的社会原型。

### 推荐的原型集合

- 管仓者：负责粮食发放、库存关注、容易卷入争议
- 工匠：负责修理工具，常被多方请求插队
- 值守者：负责山口、夜巡、异常报告
- 农务主力：承担大部分稳定生产任务
- 跑腿/传话者：高移动频率，天然制造信息传播链
- 照料者：对伤病、儿童、老人等事务敏感
- 村中权威者：分配任务、裁决冲突，但未必事事都能及时决断

### 一个重要原则

这些原型应当是“常见责任中心”，而不是刚性的职业锁。

也就是说：

- 某个农务主力临时也可以去守门
- 某个跑腿者也可能临时看管工具
- 某个照料者也可能参与传话与调度

这样世界才会出现真正的再分工，而不是脚本化履职。

## 让世界活起来的变化源

山谷小村的变化源应该以“小扰动、高连锁”为主，而不是大型灾难。

### 推荐的变化源池

- 一件关键工具损坏
- 一位重要劳动力缺席
- 谷仓记录与实物不符
- 山口来了陌生旅人
- 某条边暂时封闭或难以通过
- 夜间有人被看见走了后坡小路
- 一则未经证实的传闻开始传播
- 降雨或降温改变当天优先级

这些变化源的共同特点是：

- 单个事件不需要复杂物理模拟
- 但会迅速改写多个人的计划和观察

## 规则引擎至少要表达哪些事实

为了让这个场景真正可运行，世界真相层至少要有以下几类事实。

### 1. 空间事实

- 地点集合
- 连接边集合
- 每条边的通行性、可视通性、耗时、条件

### 2. 在场事实

- 哪个 Agent 当前位于哪个地点
- Agent 是否处于移动中，目标地点是什么

### 3. 资源事实

- 某个关键物品当前在谁手里，或位于哪个地点
- 某类资源的库存是否充足

MVP 即便暂不做通用容器系统，也至少要能表达：

- 斧头在工坊
- 水桶由某人持有
- 粮食在谷仓库存中

### 4. 任务与承诺事实

- 某个 Agent 是否已经接受某项任务
- 某项任务是否有完成时限
- 某项任务是否依赖前置条件

### 5. 事件事实

- 未来动作何时出队
- 它依赖哪些世界前置条件
- 失败时应返回哪类结果

### 6. 观察分发事实

- 哪些 Agent 应当获知某事件
- 是立即获知，还是通过他人转述延迟获知

## 讲述者应该强调什么

Narrator 在这个场景里不应只写氛围，而应尽量稳定地产出以下几类信息：

- 你所在地点和可达地点
- 你可直接观察到的人员、资源、异常
- 最近刚发生的变化
- 当前明显可做的几类行动
- 哪些信息是传闻、猜测或未核实

理想状态下，Narrator 输出的观察文本应让 Agent 明白：

- 我现在在哪
- 我知道了什么新变化
- 我接下来有哪几类可行动作
- 哪些事情具有时间压力

## 首批高价值情境

以下情境故意围绕“地点图 + 事件队列”设计。

### 1. 单工具冲突

- 村里只有一把能用的斧头
- 两个人几乎同时计划使用它
- 先到者成功，后到者失败并需要重规划

### 2. 路径回避

- 某人不想穿过中央空地被别人看见
- 选择后坡小路绕到住屋区
- 另一个巡查者是否能及时发现，取决于其当前路径和视野覆盖

### 3. 送达错位

- 任务要求把修好的工具送到工坊外的接收者手中
- 但接收者中途改道去了农田
- 送达者抵达后只得到“目标不在”的结果

### 4. 山口来客

- 陌生旅人出现在山口哨点
- 守卫、村中权威者、工匠、管仓者对其反应不一定一致
- 来客既可能带来交易，也可能带来谣言和风险

### 5. 夜间传闻

- 有人声称看见另一个人夜里去了谷仓方向
- 这一观察可能真实，也可能是误认
- 消息扩散先于核查发生

### 6. 临时补位

- 原定要去取水的人突然病倒
- 谁来补位会影响工坊和住屋区的后续活动

## 为什么这个场景会有“1 + 1 > 2”

单独看，每个系统都不复杂：

- 地点图不复杂
- 资源种类不复杂
- 角色数量不复杂
- 事件语义也不复杂

但它们组合后会自然生成比单项复杂度更高的现象：

- 因为空间分离，所以消息传播会滞后
- 因为资源稀缺，所以任务会互相打断
- 因为人物重复相遇，所以失败会沉淀成关系变化
- 因为存在回路和低可见路径，所以追逐和回避有意义
- 因为事件出队时重验，所以计划会不断被现实修正

这就是这个场景最有价值的地方：

- 它不靠预写剧情出故事
- 而是靠一个小社会持续运转，自己长出故事

## 对实现路线的直接启发

如果以山谷小村为首发场景，那么规则引擎的实现顺序可以更明确：

1. 先把地点图、Agent 位置、移动耗时做出来
2. 再把关键物品/库存位置和占用关系做出来
3. 再把事件队列与出队重验做出来
4. 再把观察分发与传闻延迟做出来
5. 最后才考虑地点内锚点、容器层级、复杂天气等增强项

这个顺序比较稳，因为每一步都能被前面的 user story 直接验证。

## 当前不急着做的东西

为了保护 MVP，以下内容当前不应成为山谷小村首版的阻塞项：

- 地点内格子与朝向
- 家具与房间内部几何
- 精细战斗系统
- 复杂市场价格模型
- 高保真历史考据
- 完整血缘/婚姻/家谱系统

这些都可能有价值，但现在引入会稀释主链。
