# SimWorld 主体画像

> 用途：回答“谁在 SimWorld 里生活”这个比场景皮肤更底层的问题，并给出比“默认都是人类”更稳的抽象。

## 结论先行

SimWorld 不应把核心主体硬编码为“人”。

更合适的做法是：

- 把世界的空间、时间、资源、事件因果作为相对稳定的外部骨架
- 把生活在其中的主体定义为可切换的 inhabitant profile

换句话说，山谷小村首先是一个社会与因果结构，不是一个“必须住着生物学人类”的设定。

当前更推荐的路线是：

- 世界先做人类社会友好的因果骨架
- 首个主体 profile 采用 human-like，但不要做 full human simulation
- 后续再在同一骨架上替换为 household-robot、戏剧角色或混合主体

## 先分清两个问题

这个问题经常会缠在一起，但其实应该拆开：

### 1. 世界像什么

例如：

- 山谷小村
- 海湾聚落
- 驿站社会

这是空间结构、资源循环、社会组织的问题。

### 2. 谁生活在里面

例如：

- Human-Lite 村民
- Household-Robot-Lite 居民
- 戏剧化角色主体
- 混合社会中的多类主体

这是 needs、risk、motivation、norms、narrative lens 的问题。

把这两个问题拆开之后，很多设计会立刻变清楚：

- 山谷小村并不必然意味着“完整的人类模拟”
- 机器人主体也不必然要求“科幻空间站”，同样可以住进一个有地点图、有协作压力的小社会骨架里

## 为什么不应默认“人类”

### 1. “人类”不是中性默认值，而是一整套很重的建模承诺

一旦把主体默认成人类，系统很容易被拖向：

- 饥饿
- 疼痛
- 疲劳
- 睡眠
- 受伤恢复
- 家庭结构
- 面子与羞耻
- 爱恨情仇

这些要素当然都可能有价值，但它们不是一个整体，没必要一次性全部绑进首版。

### 2. 当前 LLM 的出厂人格与“真实的人”并不一致

现代通用 LLM 的行为先验，往往是被在线问答与工具调用场景塑形的。

它们常见的默认倾向包括：

- 过度解释
- 过度合作
- 缺少持续自发活动
- 很弱的长期欲望
- 不自然地回避主见、冲突与关系黏性

如果直接要求这类模型“去当一个真实的人”，经常会得到一种既不像人、也不像工具、还不稳定的中间状态。

因此更稳的路线不是硬演“完整人类”，而是明确规定：

- 这个主体到底需要什么
- 它怕什么
- 它会为什么事情优先级升高
- 它应如何理解与他人的关系

### 3. 未来目标可能根本不是“像人”，而是“像某类 agent”

如果未来目标是家用机器人，那么关键约束可能是：

- 电量管理
- 维护窗口
- 碰撞厌恶
- 用户安全优先
- 权限与可操作范围

这和人类主体并不一致。

反过来，如果目标是影视或游戏角色驱动，那就可能更强调：

- 情绪波动
- 关系张力
- 价值冲突
- 戏剧性选择

所以“默认人类”并不能同时服务这两类目标。

## 为什么又不建议一开始彻底去人类化

虽然不该把主体写死为人，但首个主体 profile 仍然应该 human-like。

原因很现实：

### 1. LLM 对人类社会的文本先验最强

诸如这些东西，模型通常都更熟：

- 借还
- 承诺
- 轮班
- 分工
- 偷懒
- 争执
- 通报
- 求助

所以首个场景若要快速长出稳定行为，human-like social baseline 是最顺手的。

### 2. 山谷小村本身就是人类社会语法友好的骨架

这个场景最擅长表达的是：

- 谁该做什么
- 谁先到达
- 谁打断了谁
- 谁该补位
- 谁掌握了资源

这些都是 human-like social coordination 的强项。

### 3. “Human-Lite” 比 “完整的人” 更适合作为第一种主体

第一种主体不必是 full human simulation，而可以是一个裁过边的 profile：

- 保留人类社会协作的骨架
- 弱化不必要的生理与心理细节
- 只保留会直接影响行动规划与事件因果的部分

## 建议的抽象：Inhabitant Profile

建议把主体类型收敛成一个明确的 profile，而不是散落在 prompt 里的角色扮演描述。

### 一个 profile 至少应包含五层

#### 1. Embodiment

主体如何在世界中存在与行动：

- 移动方式
- 可携带能力
- 感知范围
- 脆弱性类型
- 恢复或维护方式

#### 2. Need Channels

主体会被哪些“持续性需求”驱动：

- 食物
- 水
- 休息
- 电量
- 维护
- 温度安全

这些需求不一定都要存在，但必须是显式可定义的。

#### 3. Appraisal Channels

主体如何感到“事情不妙”或“事情重要”。

与其把这层写死成“人类情绪”，不如先抽象成 appraisal：

- 紧迫
- 危险
- 匮乏
- 损伤风险
- 社会债务
- 可疑性

对于不同主体，这些通道可以映射成不同体验：

- 人类：疲劳、疼痛、羞耻、恐惧
- 机器人：低电量、碰撞告警、过热、维护超期
- 戏剧角色：嫉妒、愤怒、希望、羞辱

#### 4. Social-Norm Model

主体如何理解与他人的关系：

- 是否倾向服从权威
- 是否重视互惠
- 是否保守或冒险
- 是否默认共享信息
- 是否在乎声誉与承诺

#### 5. Narrative Skin

同样的内部状态，如何被叙述成观察与内心文本。

例如：

- Human-Lite 会更像“累、饿、担心、没面子”
- Household-Robot-Lite 会更像“低电量、碰撞风险升高、维护窗口不足”

这层更适合放在 narrator 与 prompt，而不是世界真相层。

## 哪些东西必须进规则引擎，哪些可以留给 prompt

这是一个关键分界。

### 必须由规则引擎掌握的

- 电量/食物/水/休息等硬需求状态
- 碰撞风险、损伤状态、维护状态
- 权限、禁区、可执行动作范围
- 行动耗时与前置条件
- 资源占用和持有关系

这些一旦只靠 prompt，就很容易漂。

### 可以由 profile 配置 + prompt 激活的

- 是否更谨慎或更冒险
- 是否更重承诺
- 是否更乐于协作
- 说话风格与内心叙述风格
- 对某些关系的情感着色

换句话说：

- hard constraints 进世界模型
- soft stance 进 profile 与上下文

## 对山谷小村的直接建议

### 1. 先不要把“村民”理解成生物学人类

更稳的理解应是：

- 村中主体是承担社会角色的一组 inhabitants
- 当前只是先采用 human-like social grammar 来设计世界

也就是说：

- “管仓者”首先是一种责任中心
- “值守者”首先是一种岗位功能
- “工匠”首先是一种技能与资源耦合点

它们不必自动等于完整人类建模。

### 2. 首个推荐 profile：Human-Lite Villager

建议第一种主体 profile 具备：

- 会移动、会等待、会错过、会传话
- 理解承诺、排队、插队、借还、求助
- 有粗粒度的体力/休息压力
- 有粗粒度的受伤/不适风险
- 在乎任务完成、关系评价和公开失败

但暂不要求：

- 高保真疼痛体验
- 复杂恋爱、婚姻、家谱
- 高保真生理细节
- 过重的情绪戏

### 3. 第二种推荐 profile：Household-Robot-Lite

如果未来要面向家用机器人，这个 profile 很值得尽早预留。

它可以具备：

- 需要规划充电时间与充电地点
- 高碰撞厌恶与安全优先
- 没有饥饿与疼痛
- 通过维护与损伤，而不是自愈，维持长期状态
- 对权限、主人指令、禁区更敏感

它和 Human-Lite 的最大差异，不在“会不会说话”，而在：

- 什么构成持续压力
- 什么构成高优先级中断
- 什么构成不可接受风险

### 4. 第三种 profile：Dramatic Character

如果未来要走影视/游戏角色驱动，可以额外定义：

- 更显性的情绪通道
- 更高的关系张力
- 更强的个人执念
- 更强的戏剧性偏好

这类主体并不适合作为 SimWorld 的第一默认值，但很适合作为后续对比 profile。

## 一个很有价值的研究方向：骨架复用，画像切换

这可能是 SimWorld 最有潜力的一点。

同一个山谷小村骨架，可以尝试跑多种主体 profile：

- Human-Lite
- Household-Robot-Lite
- Dramatic Character

如果世界的地点图、事件链、资源竞争结构保持相近，那么我们就能观察：

- 哪些行为模式是由因果结构决定的
- 哪些行为模式是由主体画像决定的
- 哪些 post-training 数据在跨画像时仍有迁移价值

这比一次只做一种“固定人格设定”的世界，更适合作为后续小模型训练与诊断平台。

## 关于“第一系统”与情绪注入

你提到“第一系统注入喜怒哀乐等本能”，这个方向是有价值的，但我不建议一上来就直接做成人类情绪模拟器。

更稳的做法是：

- 先定义 appraisal channels
- 再让不同 profile 把这些通道渲染成不同的主观体验或叙述风格

这样有两个好处：

- 统一底层结构，避免每种主体都从零写一套心理系统
- 允许少量数据跨画像迁移，因为“紧迫/损伤风险/社会债务”这类骨架是共通的

## 当前主张

如果要明确表态，我的主张是：

- SimWorld 的世界骨架不应默认“以人类为唯一主体”
- 但 SimWorld 的第一个主体 profile 应该是 Human-Lite，而不是纯机器人或纯戏剧人格
- 山谷小村当前最适合作为“human-shaped social coordination substrate”
- 后续应尽早验证在同一骨架上切换 Household-Robot-Lite 的可行性

这条路线比较稳，因为它同时兼顾：

- 当前 LLM 的先验优势
- 未来 Agent 类型的可扩展性
- 少量数据跨画像泛化的可能性
