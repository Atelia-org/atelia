# 近程里程碑：用主流 LLM 产出非对话范式训练语料

> 用途：为 Atelia 当前阶段定义一个更近、更稳、且能直接牵引 SimWorld / Agent.Core / Completion 协同收敛的里程碑目标。

## 一句话主张

当前 Atelia 更合适的近程里程碑，不是“直接训练出强自主 Agent”，也不是“先做 RL 奖励闭环”，而是：

- 用主流 LLM 在明确约束下生成高质量角色分析语料
- 把这些语料变换为一种非对话范式的小模型训练数据
- 由此先塑造一种不同于 chat assistant 的心智模板与行动接口

这个里程碑的价值在于：

- 它足够近，可以用现有 LLM 和当前工程骨架推进
- 它足够硬，能倒逼 SimWorld、Agent.Core、Completion 形成真正服务训练数据生产的统一接口
- 它避开了太早进入 RL、奖励设计、benchmark 刷分这些高成本分支

## 为什么这件事值得先做

### 1. 它比“直接做终局 Agent”更可控

如果当前目标直接设成“做出长期连续自主 Agent”，工程上会立刻被拉向很多不稳定问题：

- 目标管理
- 长期记忆污染
- 奖励设计
- 在线探索
- 安全边界
- 失控恢复

这些都不是不重要，但它们太容易把当前阶段的努力耗散掉。

而“先做高质量语料，用来训练一种非 chat 小模型”有更清晰的收敛面：

- 输入模板是什么
- 输出模板是什么
- 哪些行为是期望的
- 哪些错误可以自动检测
- 数据好坏如何比较

### 2. 它比 RL 更适合当前阶段

你说的“更像一种带变换的蒸馏”这个判断我基本赞同。

当前更像是在做：

- 世界驱动的数据采样
- 角色化视角变换
- 推理/行动轨迹蒸馏

而不是：

- 基于奖励探索策略空间
- 追求泛化到开放任务的最优策略

这条线的直接好处是：

- 能更快开始
- 数据质量问题更容易被肉眼和规则共同审查
- 失败模式更容易定位在模板、prompt、世界模型还是数据过滤上

### 3. 它不以“刷分”或“专业技能”作为近期目标

这其实很重要。

如果近期目标是：

- 数学分数更高
- 编程更强
- 专业考试更好

那整个系统很容易被牵着走回 chat assistant 的主赛道。

但你这里提的目标更像是：

- 创造一种不同的默认心智模型
- 创造一种不同的行动接口
- 创造一种不同的世界理解方式

这反而更符合 Atelia 的独特价值。

## 我对这个里程碑的判断

如果要明确表态，我会支持把它作为当前 Atelia 的近程主里程碑。

但我会把措辞再收紧一点：

> **近程里程碑**：构建一条稳定的数据生产链，使用主流 LLM 在 SimWorld 中以角色分析模式生成高质量轨迹，并将其转换为“Observation + Belief -> Reasoning + Tool-Call”范式的小模型训练语料。

这样写有两个好处：

- 它强调“数据生产链”是交付物，而不只是一个训练想法
- 它把 SimWorld、模板、轨迹转换、质量控制都纳入范围

## 所谓“非对话范式”在这里具体意味着什么

你给的定义已经很接近可实现形式了。我这里把它再压缩成一个更可操作的版本。

### 传统 chat 范式

大致是：

- `system`
- `user`
- `assistant`

训练和推理默认都围绕“回答用户”展开。

### 当前想要的非对话范式

更像是：

- `Belief`
- `Observation`
- `Reasoning`
- `Tool-Call`

其中：

- `Belief` 取代 `system prompt` 的很多职责，但不再只是“平台规则说明”，而是主体当前的内在世界模型与立场基座
- `Observation` 取代 `user`，代表来自外部世界的可感知输入
- `Reasoning` 不再是为了写给人看的解释，而是主体内部的 deliberation 痕迹
- `Tool-Call` 成为唯一外显动作接口

### 一个关键特征：没有 `response`

这是个很强的选择，我认为是对的。

它意味着：

- 模型不以“回复某个人”为默认目的
- 模型的输出不是社交性文本，而是内部推理与行动发起
- 如果需要说话、汇报、请求、警告，这些都应作为世界中的动作，通过 tool-call 表达

例如：

- `SpeakTo(target, content)`
- `ReportTo(target, summary)`
- `AskForHelp(target, reason)`

这样“说话”就从 assistant response 退回成一种世界内动作，而不是模型的存在目的。

## 为什么这会塑造不同的心智模型

这件事的关键不在“换了几个 token”，而在于：

- 训练目标不再奖励“像客服一样接话”
- 上下文结构不再把“用户提问”当成世界中心
- 输出接口不再鼓励用自然语言把一切糊过去

模型会更自然地被拉向：

- 持续观察
- 内部更新
- 风险评估
- 计划变更
- 动作发起

而不是：

- 取悦提问者
- 做解释型长回复
- 把行动冲动折叠进礼貌话术

## Belief 不应只是“换名字的 system prompt”

这里我想稍微坚持一下：

如果 `Belief` 只是把 `SystemPrompt` 改名，那意义不大。

更好的理解是：

- `Belief` 是主体当前的内在认知基座
- 它包含世界假设、角色立场、任务连续性、长期偏好、风险观、关系判断等
- 它不等于平台给的总规则说明

也就是说，`Belief` 更像：

- 当前我认为世界是什么样
- 当前我认为自己是谁
- 当前我在乎什么
- 当前我对他人和局势的默认预设是什么

而不是简单地说：

- 你必须礼貌
- 你不能做坏事
- 你是一个助手

这是你这条路线里最值得认真做的一个概念点。

## 一个需要提前防漂的点：Reasoning 不是随意散文

如果输出全部是 `reasoning + tool-call`，那么 `reasoning` 很容易重新退化成：

- 聊天式解释
- 冗长自说自话
- 模仿主流 LLM 的礼貌分析腔

所以这里最好尽早约束 `reasoning` 的角色。

我更推荐把它理解成几类结构化思维痕迹，而不是自由散文：

- 当前局势摘要
- 风险评估
- 前置条件检查
- 计划变更理由
- 行动选择依据

甚至后续可以进一步压成半结构化 block，而不是纯自然语言长段落。

否则你虽然拿掉了 `response`，模型还是会偷偷长成另一种 assistant 腔。

## 为什么不遮蔽常识知识是合理的

你说“不需要因为自己是农夫而装作不会微积分”，这个我认同。

原因有三层：

### 1. 常识知识遮蔽会制造不自然的训练目标

如果模型明明知道某些知识，却被训练成“为了角色扮演而假装不知道”，就会把大量容量浪费在：

- 何时该装不知道
- 如何演得像不知道
- 如何维持这种表演的一致性

这对当前目标帮助不大。

### 2. 你要塑造的不是职业 cosplay，而是认知与行动模板

重点不该是“它是否只会说农夫该会的话”，而应是：

- 它是否以观察为中心
- 它是否用 belief 组织行动
- 它是否把输出压到 reasoning 与 tool-call
- 它是否在世界里持续作出可验证动作

### 3. 真正该限制的不是知识，而是 affordance 与 belief

比如一个村中主体可以知道微积分，但这并不意味着：

- 它会在当前世界里主动用微积分解决取水任务
- 它具备执行超出自身 affordance 的动作能力
- 它会忽视当前 belief、风险和任务上下文

也就是说：

- 不必遮蔽知识
- 但必须限制行动接口、任务上下文和主体优先级

## SimWorld 在这个里程碑里的角色

SimWorld 在这里不只是 demo 世界，而更像语料工厂的世界引擎。

它至少承担四个作用：

### 1. 生成 Observation

世界真相经过可见性与叙述投影，形成主体可见 observation。

### 2. 承载 Belief 更新压力

因为 observation 是局部且延迟的，主体必须持续更新 belief，而不是每轮都从零回答。

### 3. 验证 Tool-Call 的真实后果

模型发出的动作不是装饰文本，而会改变世界并在后续 observation 中回流。

### 4. 生成可筛选的轨迹数据

一个 episode 可以天然包含：

- observation 序列
- belief 演化
- reasoning 片段
- tool-call 序列
- 结果事件
- 失败与修正

这正是高价值训练样本的基本形态。

## 一个更稳的最小数据管线

如果围绕这个里程碑设计，我建议先想成一条 6 段管线：

1. `SimWorld Episode Generator`
   - 生成初始世界状态与扰动
2. `Perspective Projector`
   - 为特定主体投影 observation
3. `Role Analyst LLM`
   - 用主流 LLM 生成 belief update、reasoning、tool-call
4. `World Executor`
   - 执行 tool-call 并产生真实事件
5. `Trace Filter / Critic`
   - 过滤低质量、漂移、聊天腔、工具滥用等样本
6. `Format Transformer`
   - 把轨迹转换成非 chat 小模型的训练样本

这个里程碑的交付物不该只是一个 prompt，而应是这条管线能稳定出样本。

## 我建议新增的质量标准

如果要做这个里程碑，数据质量标准最好一开始就有，不然后面会堆很多“看起来挺像”的低价值样本。

至少建议检查：

### 1. 非聊天腔

- reasoning 不应频繁出现 assistant 式礼貌回答
- 输出不应默认面向“用户”说话

### 2. 工具驱动

- 关键外部行为必须通过 tool-call 发起
- 不允许用 reasoning 假装完成世界动作

### 3. Belief 连续性

- 当前 belief 必须能被上一轮 observation 和历史事件解释
- 不应每轮重置人格与立场

### 4. 失败可见性

- 样本中应包含失败、阻塞、错过、误判与修正
- 不能只有一路顺滑成功的“完美轨迹”

### 5. 可重放性

- 同一轨迹至少在世界事件层面应能复现
- tie-break、reservation order、observation delivery 需要稳定

## 我会补充的一条边界

虽然这条路线不做 RL，我仍然建议不要把它理解成“纯静态蒸馏”。

更准确地说，它是：

- 有世界反馈的轨迹蒸馏
- 有角色化视角变换的数据变换
- 有行为过滤与风格去偏的数据工程

它比纯 SFT 更接近“行为蒸馏”，只是还没有进入 reward optimization。

## 当前建议的里程碑措辞

如果要落成项目层面的近程目标，我建议这样表述：

> **M1：建立非对话范式数据生产链。**
> 基于 SimWorld 与主流 LLM，以角色分析模式稳定生成高质量轨迹，并将其转换为 `Belief + Observation -> Reasoning + Tool-Call` 的小模型训练语料。

这个版本的优点是：

- 目标明确
- 工程边界清晰
- 不预设一定要先有 RL
- 不把“像人聊天”误当成最终形式
