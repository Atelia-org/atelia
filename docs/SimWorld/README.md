# SimWorld 初始记录

> 用途：作为 SimWorld 方向的入口文档，记录已达成共识的目标、当前工作假设、待决策问题，以及后续应拆分的专题。

## 定位

SimWorld 不是“让 LLM 即兴编剧情”的 DM 系统，而是一个面向 LLM Agent 的持久文本虚拟世界。

它的目标是：

- 为 prototypes/Agent.Core 提供可长期运行、可持续交互、可验证反馈的外部世界。
- 为后续非 Chat 范式的小型 LLM 提供可自动生成轨迹、可评测、可重放的训练与诊断环境。
- 让“叙述”退回到观察接口的位置，让世界因果、状态推进、资源流动、信息传播先独立成立。

## 当前共识

### 1. 世界先于叙述存在

规则引擎负责世界中的 hard truth：

- 空间组织
- 时间推进
- 资源与库存
- 所有权与容器
- 查询与记账
- 事件日志

LLM 主要负责：

- 把世界状态转成可感知观察
- 解释主观判断、对话、计划与意图
- 在不完整信息下作出行动决策

### 2. 必须区分三种信息层

系统至少应区分：

- 世界真相：世界里客观发生了什么
- 可感知信息：某个视点此刻能看到、听到、推断到什么
- 主体信念：角色自己以为是真的什么

这是后续实现误解、侦查、传闻、社会传播、记忆偏差的基础。

### 3. Narrator 的职责是感知压缩，而不是作者代笔

“讲述者”更像主视点的感知压缩器：

- 输入：规则引擎按视点生成的 perception packet
- 输出：适合某个 Agent 使用的观察文本

Narrator 不应直接决定世界中“真的发生了什么”。

### 4. 首个可行世界不追求图形表现

图形表现只服务于：

- 开发调试
- 状态诊断
- 必要时向多模态模型传达局部空间结构

主交互形态仍应是文本。

### 5. MVP 的空间模型先做减法

当前进一步收敛为：

- MVP 不做地点内格子
- MVP 不做地点内持续几何布局
- MVP 先只保留地点之间的连接关系
- 容器、地点内语义锚点、局部可供性图放到后续迭代

这样做不是否认它们有价值，而是承认当前更值得先攻克的问题是：

- 地图信念如何注入给 Agent
- Agent 如何在空间图上规划与移动
- 多个 Agent 之间的可见性如何判定
- 追逐、躲避、搜索这类行为如何在地点图层面成立

MVP 的目标不是还原房间里的家具布局，而是尽快得到一个能够支撑：

- 空间分离
- 空间移动
- 相遇与错过
- 追逐与逃离
- 基于视野的信息差

的最小世界。

### 6. MVP 的时间模型采用“动作耗时 + 事件队列”

当前进一步收敛为：

- MVP 不实现局部细时钟
- 所有行动通过声明耗时进入事件队列
- 世界按事件出队顺序推进，而不是按统一 tick 空转

首个重点不是“更细的时间分辨率”，而是：

- 事件顺序稳定
- 因果一致性稳定
- 前序事件对后续事件的失效影响可正确处理

## MVP 当前收敛

### 空间：地点图优先，地点内细节延后

MVP 的空间真相层先只包含：

- Location：地点
- Connection：地点之间的连接边
- EdgeState：边的状态，例如开启、关闭、上锁、阻塞

连接边至少应允许表达：

- 是否可通行
- 是否可视通
- 是否需要条件才能通过

这意味着 MVP 中的“遮挡/阻挡”主要发生在地点之间，而不是地点内部几何里。

换句话说：

- 不要求规则引擎回答“房间内沙发挡住了半个书架吗”
- 先要求它回答“门关着时隔壁房间是否可见、是否可通行”

这会刻意牺牲房间内部的精细空间博弈，但能换来一个更稳的第一阶段。

### 关于地点内语义锚点

语义锚点仍然是一个很有前途的方向，但更适合作为 MVP 之后的第一批增强项，而不是首版核心依赖。

原因很直接：

- 一旦引入地点内锚点，就会自然引出地点内可见性、可达性、占位、相对位置等一串派生问题
- 这会迅速把 MVP 从“地点图世界”推向“轻量局部空间模拟器”

因此当前建议是：

- 先把地点图跑通
- 等地图信念、移动、视野、追逐这些主链稳定后，再把锚点作为局部结构补进来

### 时间：事件出队时重验，而不是每次全队列重算

MVP 中每个未来事件都应被视为“基于当前世界状态作出的预约动作”，而不是已经承诺必然成功的未来事实。

因此执行语义应为：

- 事件入队时记录动作、目标、计划执行时间，以及必要的前置条件
- 事件真正出队时，必须基于最新世界状态重新校验前置条件
- 若前置条件不成立，则该事件转化为失败、打断、部分成功或无事发生，而不是强行执行旧效果

这能直接覆盖典型冲突：

- A 和 B 争抢同一个物品
- A 先拿到后，B 的原动作在出队时发现前置条件失效
- B 得到的结果应是“未拿到/目标已不存在/已被他人取得”，而不是复制成功

一个重要实现取舍是：

- 不建议每发生一个事件，就主动扫描并重写整个后续队列
- 更稳的做法是让后续事件在出队时按最新世界状态惰性重验

必要时，可以给事件附带它依赖的实体引用或版本戳，以便快速判断其是否已过时。

### 关于“需要重新响应”

前序事件导致后续事件失效后，系统至少要区分两件事：

- 世界层结果：原计划动作失败、被打断、或被替换为别的结果
- Agent 层反应：该主体何时收到这条新观察，并据此重新决策

MVP 可以先不自动替 Agent 生成补救动作，但必须把失败结果和新观察稳定地送回给该 Agent。

## 之前候选方案的状态更新

### 空间粒度

在更长期的方向上，“地点图 + 地点内语义锚点 + 按需展开的局部结构”仍然成立。

但对 MVP 而言，当前收敛为：

- 先只做地点图和边语义
- 不做地点内格子
- 不做地点内锚点必选项
- 不把房间内部遮挡作为首版约束重点

### 时间粒度

在更长期的方向上，混合式时间模型仍有价值。

但对 MVP 而言，当前收敛为：

- 先做动作耗时 + 事件队列
- 不做局部细时钟
- 优先把事件顺序、冲突处理、因果一致性做扎实

## 当前待决策问题

### A. 空间粒度

当前主要有两种候选：

#### 候选 1：细粒度方格子世界

特点：

- 空间关系明确，遮挡、距离、站位、路径等表达自然
- 更容易支持战术行为、局部搜索、容器摆放、可达性判断
- 更接近传统 roguelike / FF 式小地图

问题：

- LLM 可能并不擅长高频处理“绕过桌椅、转身面向、逐格移动”这类低价值空间细节
- 大量动作会膨胀为琐碎工具调用，污染上下文与轨迹质量
- 研发重点可能从“智能体与世界”偏移到“空间微操模拟”

#### 候选 2：场景/地点图世界

特点：

- 世界由相互连接的抽象地点组成，例如房间、街道、店铺、林地、塔楼
- 地点内部没有持续显式的微观站位
- 同一地点内的大部分移动默认由规则引擎吸收
- 更接近 MUD、AVG/GAL 的场景切换式交互

优势：

- 大幅降低空间推理负担
- 让 LLM 把 token 预算更多用于观察、计划、对话、社会关系与问题求解
- 更适合当前阶段快速做出一个可生长世界

风险：

- 地点内部若完全无结构，搜索、潜行、遮挡、物体相对位置等玩法可能被压扁
- 若抽象过头，世界会退化成“房间名 + 文本描述”的弱空间系统

#### 候选 2.5：Scene + 语义锚点 + 可供性图

这是当前更值得优先考虑的折中方案。

核心想法：

- 世界主结构仍然是地点图
- 但地点内部并非纯文本黑箱，而是保留少量“语义锚点”
- 例如：门口、窗边、书架、柜台、壁炉、床、后厨、楼梯口
- 行为优先针对这些锚点和可供性交互，而不是针对笛卡尔坐标

这样可以保留一部分局部空间意义：

- 哪些东西彼此相邻
- 哪些位置更隐蔽
- 哪个锚点能观察到街道
- 哪个容器位于上锁区域

同时避免把普通行动拆成逐格移动。

这类模型更像：

- Scene Graph
- Affordance Graph
- Landmark Graph

而不是完整物理网格。

#### 长期倾向：分层空间，而不是二选一

当前更倾向于采用三级抽象：

- World 层：地区、道路、聚落、地标
- Scene 层：酒馆、农田、街道、林地、房间
- Local 层：只在确有需要时，临时展开为更细的局部结构

长期初步原则：

- 默认以 Scene 层作为主要交互单位
- Scene 内部优先使用“语义锚点 + 可供性图”，而不是默认展开成格子
- 只有在“空间细节本身具有信息价值或博弈价值”时，才进入 Local 层
- 同一地点内的低价值位移由规则引擎自动吸收，不要求 LLM 逐步绕行

这意味着“检查书架”通常可以直接是一个高层动作，而不是拆成：

- 走到书架前
- 调整朝向
- 避开沙发
- 再查看第三层隔板

除非该空间本身具有明确的策略性约束，例如：

- 火场中的可通行路径
- 守卫的视线覆盖
- 只露出半边的血迹
- 某个物体被遮挡在家具之后

### B. 时间粒度

时间粒度尚未定，但它和空间粒度高度耦合。

当前至少有三种思路：

#### 候选 1：严格离散 tick

- 每一步推进固定时长
- 所有实体统一按 tick 更新
- 最易实现、最利于记账和重放

问题：

- 若动作粒度不统一，会出现大量“空转 tick”
- 社交、旅行、搜索、工作等行为很难共用一个舒服的时间步长

#### 候选 2：动作驱动的可变时长

- 每个动作声明自身耗时
- 世界时间按动作结果跳跃推进
- 更贴近文本世界中的高层行为表达

问题：

- 并发实体的调度与插队会更复杂
- 需要一个稳定的事件队列或预约系统

#### 候选 3：混合式时间

- 宏观层面用事件驱动/动作驱动推进
- 局部需要精细模拟时再退回较细 tick

长期倾向：混合式时间更匹配分层空间。

一个自然组合是：

- Scene 层主要用动作驱动的可变时长
- Local 层在需要局部博弈时启用更细时间步
- 远处世界用低保真聚合推进，而不是逐实体逐秒模拟

## 一个重要的设计主张

SimWorld 的目标不是最大化“空间真实感”，而是最大化“可产生持续因果与社会涌现的有效复杂度”。

因此应优先保留：

- 稀缺资源
- 地点连通性
- 所有权与访问权限
- 可感知与不可感知边界
- 事件传播与信念差异

而应尽量压缩：

- 低价值逐格移动
- 纯姿态微操
- 缺乏信息增益的路径细节
