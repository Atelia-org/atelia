## Pair-Function



 这是个经典问题，在空间索引（Morton code / Z-order curve）领域已经被研究透了。有两个比查表更快的方法：

## 方法一：Binary Magic Numbers（掩码分治法）

将 32 bit 展开为 64 bit，每个原始 bit 之间插入一个 0：

```csharp
static ulong SpreadBits(ulong x)
{
    x = (x | (x << 16)) & 0x0000_FFFF_0000_FFFF;
    x = (x | (x <<  8)) & 0x00FF_00FF_00FF_00FF;
    x = (x | (x <<  4)) & 0x0F0F_0F0F_0F0F_0F0F;
    x = (x | (x <<  2)) & 0x3333_3333_3333_3333;
    x = (x | (x <<  1)) & 0x5555_5555_5555_5555;
    return x;
}
```

原理是分治：先把高 16 bit 和低 16 bit 拉开 16 位的间距，再把每 8 bit 拉开 8 位，依次递减。5 步完成，每步都是 shift + or + mask，全是 ALU 指令，无分支无访存。

以 `0b1111` 为例走一遍：
```
起始:                              0b1111
x | (x << 16):  不变（值太小，高位无影响）
x | (x <<  8):  不变
x | (x <<  4):  0b1111_0000 | 0b1111 → mask → 0b0000_1111_0000_1111  (不对，值太小其实不变)
x | (x <<  2):  0b11_1100 | 0b1111 = 0b11_1111 → & 0x33 → 0b0011_0011
x | (x <<  1):  0b0110_0110 | 0b0011_0011 = 0b0111_0111 → & 0x55 → 0b0101_0101 ✓
```

## 方法二：BMI2 `PDEP` 指令（单条指令）

如果目标平台是 Haswell+ (2013) 的 x86-64，**一条指令搞定**：

```csharp
using System.Runtime.Intrinsics.X86;

static ulong SpreadBits(ulong x)
{
    // PDEP: 把 x 的每一位按 mask 中为 1 的位置散布
    return Bmi2.X64.ParallelBitDeposit(x, 0x5555_5555_5555_5555);
}
```

`PDEP(src, mask)` 的语义是：从 `src` 的最低位开始取 bit，依次填入 `mask` 中为 1 的位置。`0x5555...` 的 mask 就是偶数位全 1，正好把每个 bit 放到偶数位上，奇数位自动为 0。

反向操作（压缩，去掉中间的 0）是 `PEXT`：
```csharp
static ulong CompactBits(ulong x)
{
    return Bmi2.X64.ParallelBitExtract(x, 0x5555_5555_5555_5555);
}
```

| 方法 | 指令数 | 延迟 | 备注 |
|:-----|:------|:-----|:-----|
| 4-bit 查表 | ~N/4 次查表+移位+拼接 | 高 | 有 cache miss 风险 |
| 掩码分治 | 15 条 ALU | ~5 cycle | 全平台通用 |

## 建议

在 SJValue 的场景下，如果你需要这个操作（比如把两路数据交织编码），推荐：

```csharp
[MethodImpl(MethodImplOptions.AggressiveInlining)]
static ulong SpreadBits(ulong x)
{
    if (Bmi2.X64.IsSupported)
        return Bmi2.X64.ParallelBitDeposit(x, 0x5555_5555_5555_5555);

    // fallback: 掩码分治
    x = (x | (x << 16)) & 0x0000_FFFF_0000_FFFF;
    x = (x | (x <<  8)) & 0x00FF_00FF_00FF_00FF;
    x = (x | (x <<  4)) & 0x0F0F_0F0F_0F0F_0F0F;
    x = (x | (x <<  2)) & 0x3333_3333_3333_3333;
    x = (x | (x <<  1)) & 0x5555_5555_5555_5555;
    return x;
}
```

JIT 会在运行时消除掉 `IsSupported` 分支，不会有运行时开销。

 ## 1. 掩码分治的逆运算（Compact / De-interleave）

把交错的偶数位提取回来，就是 Spread 的镜像操作——方向反过来，先 mask 再右移合拢：

```csharp
// 逆运算：从交错编码中提取偶数位 (bit0, bit2, bit4, ...)
// 0a0b0c0d → abcd
static ulong CompactBits(ulong x)
{
    x &= 0x5555_5555_5555_5555;           // 丢弃奇数位
    x = (x | (x >>  1)) & 0x3333_3333_3333_3333;
    x = (x | (x >>  2)) & 0x0F0F_0F0F_0F0F_0F0F;
    x = (x | (x >>  4)) & 0x00FF_00FF_00FF_00FF;
    x = (x | (x >>  8)) & 0x0000_FFFF_0000_FFFF;
    x = (x | (x >> 16)) & 0x0000_0000_FFFF_FFFF;
    return x;
}
```

配套的完整交错/解交错对：

```csharp
static ulong Interleave(uint a, uint b)
{
    return SpreadBits(a) | (SpreadBits(b) << 1);
}

static (uint a, uint b) Deinterleave(ulong z)
{
    return ((uint)CompactBits(z), (uint)CompactBits(z >> 1));
}
```

BMI2 版本就是一条 `PEXT`：
```csharp
static ulong CompactBits(ulong x) => Bmi2.X64.ParallelBitExtract(x, 0x5555_5555_5555_5555);
```

## 2. 其他"两个小数→一个小数"的可逆方案

你的核心需求非常精确：**两个"大部分时候很小"的非负整数 → 一个尽量小的非负整数**，且可逆。这其实是一个经典的 **pairing function** 问题。我按"packed 结果的紧凑度"排序：

### 方案 A：Szudzik 配对函数（最紧凑）

```
pair(a, b) = a² + a + b,  if a ≥ b
           = b² + a,      if a < b
```

**性质**：这是 N×N → N 的**双射**，即每个自然数恰好对应一对。输出大小约 `max(a,b)²`。

```csharp
static ulong SzudzikPair(ulong a, ulong b)
{
    return a >= b ? a * a + a + b : b * b + a;
}

static (ulong a, ulong b) SzudzikUnpair(ulong z)
{
    ulong s = (ulong)Math.Sqrt(z);     // isqrt
    ulong remainder = z - s * s;
    return remainder < s
        ? (remainder, s)       // 原来 a < b 分支
        : (s, remainder - s);  // 原来 a >= b 分支
}
```

**优点**：理论最优紧凑度——没有浪费任何 codepoint。
**缺点**：需要整数平方根（`isqrt`），编解码有乘法/除法。

### 方案 B：Cantor 配对函数

```
pair(a, b) = (a + b)(a + b + 1) / 2 + b
```

输出大小约 `(a+b)²/2`，比 Szudzik 略大（当一个值远大于另一个时不如 Szudzik 紧凑）。也需要 `isqrt` 解码。不推荐选它，Szudzik 全方位更优。

### 方案 C：Bit 交错（Morton / Z-order）

就是我们讨论的方案。输出的 bit 宽度 = `2 × max(bitwidth(a), bitwidth(b))`。

**对比 Szudzik**：

| a, b | Szudzik | Morton |
|:-----|:--------|:-------|
| (3, 5) | 28 | 0b00_01_01_11 = 39 |
| (1, 1) | 3 | 0b11 = 3 |
| (0, 100) | 10000 | ~交错后 13-14 bit |
| (100, 0) | 10100 | ~交错后 13-14 bit |

Morton 不是双射（有些 codepoint 浪费了），但纯 bit 操作，无乘除。

### 方案 D：VarInt 拼接（Elias gamma 前缀 + payload）

思路：先用 Elias gamma 编码其中一个数的长度，然后拼接两个 payload。可以做到接近信息论最优（`bits(a) + bits(b) + O(log(bits(a)))`），但编解码是串行的、不对称的，更适合流式场景。

## 对比总结

| 方案 | 输出大小 | 编码 | 解码 | 适合 SJValue？ |
|:-----|:---------|:-----|:-----|:--------------|
| **Szudzik** | `≈ max(a,b)²` | 1× mul + 比较 | isqrt + 1× mul | ⭐ 最紧凑，解码稍慢 |
| **Morton 交错** | `2 × max_bits` | 5× ALU 或 PDEP | 5× ALU 或 PEXT | ⭐ 全 bit 操作，解码快 |
| 固定 bit-field | 固定宽度 | shift+or | shift+mask | 当前 SizedPtr 做法，不适应"通常很小" |
| VarInt 拼接 | 接近信息论最优 | 串行 | 串行 | 不适合固定宽度 handle |

## 我的判断

**对于 SJValue 的 Tagged-Pointer 场景，Morton 交错是最佳选择**，理由：

1. **与 `LeadingZeroCount` tag 方案天然契合**：两个值都小 → 交错后高位全 0 → LZC 大 → Tag 空间更充裕，完美利用你的 Tag-Code 设计。

2. **无乘除**：Szudzik 的 `isqrt` 虽然不慢，但在热路径上不如纯 bit 操作确定性好。

3. **对称**：两个维度地位平等，不需要知道"谁更大"才能高效编码。

4. **Szudzik 的紧凑优势在 64-bit handle 场景下不显著**：Morton 最多比 Szudzik 多用 1 bit（`2×max_bits` vs `2×max_bits`），但 Szudzik 需要乘法和整数平方根，trade-off 不划算。

如果你最终采用 Morton 方案来内联 SizedPtr 类型的二元组，编码后 LZC 自然反映了"两个值中较大那个的量级"，与你的 tag 系统天然配合。

## 一、补充的待决策点

你的汇总文档列了 3 大决策方向，但对话中实际已经暴露了更多需要显式决定的点：

### 1.1 `-0.0` 归一化

Lossy-Double 的 `EncodeLossyDouble` 对 `+0.0` 和 `-0.0` 会产生**不同编码**（`0x8000…0000` vs `0xC000…0000`）。你需要决定：

| 选项 | 语义 | 影响 |
|:-----|:-----|:-----|
| **保留区分** | `-0.0 ≠ +0.0`，IEEE 754 严格 | Set/Dict 中它们是两个不同的键 |
| **归一化为 `+0.0`** | `-0.0 → +0.0`，数值语义 | 编码前加一步 `if (bits == 0x8000…) bits = 0`，破坏 bitwise round-trip |

如果你选择了"跨 int/double 统一编码"，那 `-0.0` **必须**归一化为 `+0.0`（否则 `int(0)` 和 `double(-0.0)` 的编码不一致，破坏了"数值等价"的承诺）。

### 1.2 NaN 策略

Round-to-Odd 编码对 NaN payload 是有损的（多个 NaN 编码可能坍缩为同一个值）。需要决定：

| 选项 | 说明 |
|:-----|:-----|
| **单 NaN** | 所有 NaN 归一化为 canonical quiet NaN，编码前替换 |
| **保留 quiet NaN payload** | 保留 quiet NaN 的 payload bits（51 bit → 25 bit 有损） |
| **禁止 NaN 存储** | NaN 值在写入时抛异常 |

对 Agent 场景，**禁止 NaN** 或 **单 NaN** 最干净。NaN payload 在实际业务中几乎没有语义用途。

### 1.3 Subnormal Double 处理

Subnormal float（`exp == 0, mantissa ≠ 0`）的值域是 `±5 × 10^-324` ~ `±2.2 × 10^-308`。Lossy-Double 的 Round-to-Odd 对它们仍然适用（只是精度更低），但问题是：

- Agent 场景中 subnormal 几乎不可能出现
- 它们的存在使 `SJValue` 的值域更复杂
- 考虑是否应该在编码入口将 subnormal 静默 flush-to-zero？

### 1.4 Slot Index 的统一 vs 分区

你的 LZC 编码表中，每个 Slot 类型（`Slot→Double`, `Slot→String`, `Slot→DurableObject`...）都有独立的 LZC 值。但 slot index 背后的 SlotTable 结构有两种选择：

| 选项 | 说明 | 优势 | 代价 |
|:-----|:-----|:-----|:-----|
| **统一索引空间** | 所有类型共享一个连续的 slot index | 简单，index 全局唯一 | 需要二级查表取类型 |
| **分区索引空间** | 每个 Kind 有独立的 index 空间 | Kind 直接选数组，index 就是下标 | 多套分配器 |

你在对话中提过"Kind 直接就是数组选择器，slot index 直接就是数组下标"——这指向**分区方案**。值得显式写入决策文档。

### 1.5 Inline 类型的 Slot 降级版本的编码关系

每个 inline 类型都有对应的 slot 版本（超出 inline payload 时降级）。需要决定：

- inline 和 slot 版本是**同一种类型的不同编码**（Kind 相同，通过某种标志位区分）？
- 还是**不同的 Kind**（如你当前的设计：`LZC=1` 是 Inline-NonNeg-Int，`LZC=35` 是 `Slot→Full-Int64`）？

当前方案用不同 LZC 值，这意味着类型判断可能需要 `if (lzc == 1 || lzc == 35)` 来匹配"所有整数类型"。考虑是否需要一个 `KindCategory` 的快速映射。

### 1.6 确定性编码的 Round-Trip 保证等级

你提到"确定性编码，每一个值都有唯一的二进制编码"。但 Lossy-Double 引入了有损编码，这意味着：

```
Encode(1.1) → X
Decode(X) → 1.0999...998  (≠ 1.1)
Encode(Decode(X)) → X     (✅ idempotent)
```

需要显式声明的保证是什么？建议区分三个等级：

| 等级 | 保证 | 适用类型 |
|:-----|:-----|:---------|
| **Exact Round-Trip** | `Decode(Encode(v)) == v` | 整数、Boolean、Null、Char |
| **Idempotent Encoding** | `Encode(Decode(Encode(v))) == Encode(v)` | Lossy-Double |
| **Semantic Equivalence** | `|Decode(Encode(v)) - v| ≤ 1 ULP` | Lossy-Double |

## 二、建议的决策组合（三套候选方案）

根据你的约束条件和目标，我给出三套内聚的决策包：

### 方案 A：「纯数值主义」

| 决策维度 | 选择 |
|:---------|:-----|
| 跨 int/double 统一编码 | **是**，`Encode(42) == Encode(42.0)` |
| Tagged-Pointer vs NaN-Boxing | **Tagged-Pointer (LZC)** |
| Lossy-Double | **采用**，`>>1 + Round-to-Odd` |
| `-0.0` | **归一化为 `+0.0`** |
| NaN | **禁止存储**，写入时抛 `ArgumentException` |
| Subnormal | **Flush to zero** |
| 浮点数最短编码 | **不采用**（所有浮点统一走 Lossy-Double，不按 fp16→fp32→fp64 阶梯） |
| Int/Double 重叠区的规范形式 | **整数形式优先**（`42.0` 编码为整数 `42`） |
| 字典排序编码 | **不采用**（比较走解码后的语义比较） |

**优势**：语义最干净，`SJValue` 就是一个「抽象数值」。Hash/Set/Dict 对数值等价完全透明。

**代价**：写入 double 时需要先检查 `IsExactIntegerBits()`；破坏了 bitwise round-trip（`double(42.0)` 写进去再读出来变成了 `long(42)`）；需要额外类型信息才能"还原"为 double。

**适用场景**：如果 StateJournal 的语义是"存状态快照，值相等就是相等"。

### 方案 B：「类型保真主义」

| 决策维度 | 选择 |
|:---------|:-----|
| 跨 int/double 统一编码 | **否**，`int(42)` 和 `double(42.0)` 是不同编码 |
| Tagged-Pointer vs NaN-Boxing | **Tagged-Pointer (LZC)** |
| Lossy-Double | **采用**，`>>1 + Round-to-Odd` |
| `-0.0` | **保留区分**（IEEE 754 严格） |
| NaN | **单 NaN 归一化** |
| Subnormal | **保留**（通过 Lossy-Double 自然处理） |
| 浮点数最短编码 | **不采用** |
| 字典排序编码 | **不采用** |

**优势**：类型信息完整保留，`ReadAsDouble()` 和 `ReadAsInt()` 语义清晰，不会意外类型转换。

**代价**：`dict[42] != dict[42.0]`——用户需要自己注意类型一致性。diff 算法需要考虑"值相同但类型不同"算不算变更。

**适用场景**：如果 StateJournal 与 C#/.NET 类型系统强绑定，需要 round-trip 保留源类型。

### 方案 C：「务实折中」（推荐）

| 决策维度 | 选择 |
|:---------|:-----|
| 跨 int/double 统一编码 | **仅做单向归一化**：整数形式的 double 自动提升为整数编码，但整数不会被降级为 double |
| Tagged-Pointer vs NaN-Boxing | **Tagged-Pointer (LZC)** |
| Lossy-Double | **采用**，`>>1 + Round-to-Odd` |
| `-0.0` | **归一化为 `+0.0`**（一致性 > IEEE 纯粹性） |
| NaN | **单 NaN 归一化** |
| Subnormal | **Flush to zero** |
| 浮点数最短编码 | **不采用**（用 Lossy-Double 已消除 slot 回退，最短编码的优势不成立） |
| Slot 索引空间 | **按 Kind 分区** |
| 字典排序编码 | **不采用** |

**编码规则**：

```csharp
SJValue Encode(double v) {
    ulong bits = BitConverter.DoubleToUInt64Bits(v);

    // 1. NaN → canonical NaN
    if (double.IsNaN(v)) return SJValue.NaN;  // 或禁止

    // 2. -0.0 → +0.0
    if (bits == 0x8000_0000_0000_0000) bits = 0;

    // 3. 精确整数 → 提升为整数编码
    if (IsExactIntegerBits(bits)) {
        long intValue = (long)v;
        if (intValue >= 0) return EncodeNonNegInt((ulong)intValue);  // LZC=1
        else return EncodeNegInt(intValue);                           // LZC=2
    }

    // 4. 非整数 → Lossy-Double
    return EncodeLossyDouble(bits);  // LZC=0
}
```

**优势**：

- `dict[42] == dict[42.0]` 自然成立（整数形式的 double 被提升）
- 非整数的 double 仍然保留 double 类型（不会被意外当成整数）
- 编码是单调的：`Encode(x) == Encode(y)` ⟺ `x` 和 `y` 在数值上等价
- diff 算法可以直接按 bits 比较，不需要跨类型等价判断

**代价**：

- `ReadAsDouble(Encode(42.0))` 需要做一次 int→double 转换
- 写入时多一步 `IsExactIntegerBits()` 检查（但这是一个纯位运算，几纳秒级）

---

## 三、分析思路和结构性建议

### 3.1 决策依赖图

这些决策不是独立的，存在因果依赖：

```
跨int/double统一编码?
  ├─ 是 → -0.0必须归一化 → Subnormal可以flush
  │        └─ 整数形式double的规范形式必须确定（整数优先 vs float优先）
  │             └─ 编码路径需要 IsExactIntegerBits() 检查
  └─ 否 → -0.0和NaN可以独立决定
           └─ 但Dict<any,any>的跨类型去重需要用户自行处理

Lossy-Double?
  ├─ 是 → Round-Trip保证降级为Idempotent
  │        └─ 字典排序编码的吸引力降低（有损值的排序不保序）
  └─ 否 → 需要Slot回退路径
           └─ 增加序列化/diff的分支复杂度

Tagged-Pointer(LZC)?  ← 此决策已基本收敛
  └─ 是 → bit-budget按LZC递减分配
           └─ LZC 3..10的分配是高价值Decision层级决策
```

### 3.2 可逆性分析

决策的可逆程度直接影响你应该在哪些点谨慎、哪些点快速试错：

| 决策 | 可逆性 | 理由 |
|:-----|:------|:-----|
| LZC 0/1/2/62/63/64 的分配 | **不可逆** | 一旦有持久化数据，改编码表就是 breaking change |
| LZC 3..61 的分配 | **生态早期可逆** | 目前没有存量数据，可以反复调整 |
| Lossy-Double 的舍入算法 | **不可逆** | 确定性编码承诺意味着算法是语义合约 |
| `-0.0` 和 NaN 归一化 | **不可逆** | 影响 hash/equality 语义 |
| 跨 int/double 统一编码 | **不可逆** | 影响所有比较和去重逻辑 |
| Slot 分区 vs 统一 | **可逆** | 纯运行时内部实现，不影响编码格式 |
| Inline-Timestamp 的 epoch 和精度 | **不可逆** | 编码格式的一部分 |

**建议**：对不可逆决策写正式的 Decision 条款（`### decision [ID]`），对可逆决策先快速原型验证。

### 3.3 一个你可能遗漏的维度：Lexicographic Order 编码

你的附件文档最后一节提到了"直接按整数比较大小"的编码。这个思路的真正价值不在于浮点数——而在于 **SJValue 整体**。

如果所有 SJValue 都能满足 `memcmp(Encode(a), Encode(b)) == compare(a, b)`，那么：

1. **B-Tree / 排序索引可以直接用 `ulong` 比较**，不需要解码
2. **Range query 变成整数范围查询**
3. **二分查找在编码值上直接进行**

但当前 LZC 方案的自然整数序是：

```
LZC=64 (Null=0x0)  <  LZC=63 (Undef=0x1)  <  LZC=62 (Bool)  <  ...  <  LZC=0 (Double)
```

这个序没有语义意义。如果你需要 total order，需要在 LZC 分配时**刻意安排类型的排序关系**：

```
0x0000…0000              Null (最小)
0x0000…0001              Undefined
0x0000…0002, 0x0000…0003 False, True
0x0000…0004..?           其他 sentinel
≤ 0x1FFF_FFFF_FFFF_FFFF  负整数（越小的整数编码越小）
0x2000…0000..0x5FFF…FFFF 非负整数（自然序）
0x6000…0000..0x7FFF…FFFF Lossy-Double 正数
0x8000…0000..0x9FFF…FFFF Lossy-Double 负数（翻转序）
0xA000…0000..            字符串、字节串、Slot引用...
```

这是一个极有吸引力但实现复杂的方向。**建议作为 v5 的探索目标**，不要在 v4 的菱形扩张阶段引入，否则会让编码表设计的约束空间爆炸。

### 3.4 最后一个提醒：编码公式同步

tagged-pointer.md 中的 Lossy-Double 编码已经更新为带 `| 0x8000_0000_0000_0000` 的版本，但 `混杂值存储关键设计决策.md` 中仍是旧版公式（缺少 tag bit）。建议统一。

---

**总结**：我推荐**方案 C（务实折中）**作为主轴推进——Tagged-Pointer + LZC + Lossy-Double + 整数形式double提升 + `-0.0`/NaN 归一化。然后把 LZC 3..10 的分配和 Lexicographic Order 作为两个独立的后续探索方向。当前最紧迫需要"钉死"的不可逆决策是 LZC 0/1/2 的语义、Lossy-Double 的舍入算法、以及 `-0.0`/NaN 规范化规则。
