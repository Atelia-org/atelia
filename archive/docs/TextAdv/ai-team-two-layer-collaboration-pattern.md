# AI Team 双层协作模式

> 状态：已实战验证
> 适用范围：需要跨多个中等重构包、且希望减少“每一步都向用户停问”的开发/重构任务
> 目的：把这次有效的 AI Team 工作方式外置成可复用协议，避免后续会话被上下文压缩后丢失

## 一句话结论

这次最有效的工作方式，不是单个 Agent 逐文件打补丁，而是：

- 主线程负责长程目标、工作包切分、阶段闸门、跨包一致性与最终验收
- subagent 负责局部工作包的再审视、方案设计、实施落地、检查审阅、git 提交
- 主线程把 subagent 的输出当成“可审查工件”，而不是当成直接真理

这形成了一个小型的内部负反馈闭环：

- 主线程先提出候选重构方向
- explorer 再做一次局部再审视，纠偏或收敛方案
- worker 在明确边界内实施并提交
- 第二个 reviewer 从 code review 视角专挑缺口
- 主线程把 reviewer 的 finding 再压回 worker 修补

结果是：很多本来“模型能想到但常常不主动做”的分析、审阅、回归闭环，被稳定地流程化了。

## 为什么这次特别有效

核心不是“多开几个 subagent”，而是把任务层级抬高了一层。

传统单线程模式更像：

- 读一点
- 改一点
- 停一下
- 再问用户

这在小修时没问题，但在中等重构里容易出现：

- 上位目标不断被局部代码细节打断
- 明明可以内部先分析的问题，又被过早抛回给用户
- reviewer 视角缺席，导致“实现能跑，但设计收口还不够”

双层模式把这一点改成：

- 主线程先把任务拆成几个有边界的工作包
- 每个工作包内部再走一次“设计 -> 实施 -> 审阅 -> 修补”
- 用户只在更高层上参与方向确认，而不是被迫承担每一个局部判断

## 推荐角色分工

### 主线程职责

- 还原用户真实目标，而不是只看最近一句指令
- 把工作拆成边界清晰、可验收的工作包
- 决定这一轮是“先收对外协议”“先收共享骨架”“先收执行主链”，还是别的优先级
- 为 worker 设定写入范围与不该扩散的边界
- 对 explorer / reviewer 的输出做二次判断
- 决定一个包是否真的完成，还是需要追加尾修
- 维护跨包一致性，避免 A/B/C 三包各自合理但拼起来失真

### explorer 职责

- 回答局部设计问题
- 比较多种方案并给出推荐
- 标出关键文件、关键调用链、关键风险
- 不直接修改代码

### worker 职责

- 在明确边界内实施
- 补必要测试
- 运行最小必要验证
- 产出独立 commit，方便主线程按包审阅

### reviewer 职责

- 默认从 code review 心态看问题
- 优先找设计缺口、边界漏洞、测试盲区
- 不重复实现说明
- 不轻易被“测试全绿”说服

## 推荐工作流

每个工作包尽量走下面五步。

### 1. 再审视

先让 explorer 回答：

- 这个方向是否真的值得做
- 是否存在更小、更稳的切入点
- 本轮最该收什么，不该顺手收什么

如果 explorer 的结论推翻了主线程的初始判断，主线程应允许改计划。

### 2. 方案定稿

主线程根据 explorer 输出，明确：

- 本轮目标
- 明确不做的部分
- 允许写入的文件
- 推荐测试集
- 验收标准

这一步要尽量像给同事派单，而不是只说一个抽象愿望。

### 3. 实施落地

worker 负责：

- 修改代码
- 补测试
- 本地验证
- 提交独立 commit

主线程不要和 worker 并行修改同一块写入范围。

### 4. 检查审阅

让另一个 reviewer 专门回答：

- 有没有设计上没收干净的地方
- 有没有实现正确但边界不稳的地方
- 有没有“文档改了，但运行时提示没改”“写账改了，但 validator 还走旧路径”这类半收口问题

### 5. 追加尾修

如果 reviewer 提出有效 finding，主线程应：

- 明确哪些必须在本包内补上
- 哪些可以留给下一包
- 再把有效 finding 压回 worker

不要为了“保持包小”把高风险问题留到未来。

## 这次实战中验证有效的几个原则

### 原则 1

先收对外协议，再收共享骨架，最后收执行主链。

一轮中等重构之所以容易失控，常常不是因为代码太多，而是因为修改顺序反了。更稳的顺序通常是：

- 先统一对外协议，让调用方和运行时语义先说清楚
- 再收共享数据骨架或中性描述层，让多条路径先共享同一套元信息
- 最后再把执行主链向单核实现收口，减少并行 dispatcher 和语义分叉

### 原则 2

subagent 的设计结论必须经过主线程再判断。

explorer 和 reviewer 都很有价值，但它们不是自动 merge 的真理来源。主线程需要把它们放回全局路线里判断。

### 原则 3

worker 的边界要足够窄，但不能窄到无法闭环。

太宽会变成“大爆炸重构”，太窄会留下半截设计。正确做法是：一包只解决一个主问题，但允许把 reviewer 抓到的高风险尾巴补齐。

### 原则 4

reviewer 的 finding 要按风险处理，而不是按“是否方便顺手修”处理。

这类最关键的尾修，往往都来自 reviewer 发现的“半收口”问题，例如：

- 文档和 prompt 已改，但运行时仍保留旧协议提示
- 新引入的共享描述层还不是所有校验/提交路径的单一真源
- 统一执行入口表面收口了，但内部仍信任陈旧或伪造的外部输入

这些都不该留到“以后再说”。

## 推荐的主线程提示模板

给 explorer：

- 只做分析，不改代码
- 先给结论，再给依据
- 比较方案并给明确推荐
- 标注关键文件、行段、测试点

给 worker：

- 明确本轮目标
- 明确不做什么
- 明确写入范围
- 明确验收标准
- 要求独立 commit

给 reviewer：

- 以 code review 心态工作
- findings 优先
- 按严重度排序
- 若无实质性 findings，要明确写出来，并补 residual risks / testing gaps

## 何时适合用这套模式

适合：

- 中等规模重构
- 需要多轮收口的设计调整
- 有明显“协议 / 骨架 / 执行”分层的问题
- 希望减少把本可内部分析的问题过早抛给用户

不太适合：

- 单文件小修
- 明确一步到位即可完成的机械改动
- 没有自然工作包边界的超小任务

## 后续建议

以后遇到类似任务时，优先先写出：

1. 工作包列表
2. 每包的目标与禁区
3. 每包的 explorer / worker / reviewer 顺序
4. 哪些 finding 必须在本包补上

这样即使会话被压缩，主线程也能快速恢复“我们是怎么工作的”，而不只是恢复“我们改了什么”。
