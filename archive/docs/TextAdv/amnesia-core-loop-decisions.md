# TextAdv — Amnesia Core Loop Decisions

> 状态：当前方向收敛记录
> 适用范围：`prototypes/TextAdv/` 原型
> 目的：记录已经明确的目标取舍与玩法骨架，不覆盖具体字段、命令接口或规则细节

## 一句话定位

TextAdv 的首版主线，不再是“做一个普通文字冒险”，而是做一个以 `Amnesia` 为主题外壳的 Native-Agentic 训练沙盒。

它的核心任务不是提供大量剧情消费内容，而是逼迫 Player 在部分可观测、记忆受限、行动机会稀缺的条件下，持续观察、外化记忆、做出 grounded 的高层决策，并承受这些决策带来的世界后果。

## 当前已明确的决定

| 决定 | 为什么重要 | 对实现的直接含义 |
|:---|:---|:---|
| 主题采用 `Amnesia` | 让“记不清”“必须依赖外化记忆”成为世界内合理设定，而不是玩法外附加限制 | 游戏必须把记忆约束纳入核心循环，而不是当作旁支机制 |
| 时间采用离散回合制 | 直接压低工程复杂度，也更适合训练与复盘 | 世界推进应以 day/slot 为单位，而不是实时模拟 |
| 每天只给 3 到 6 次互动机会 | 稀缺行动机会会自然制造取舍 | 每回合动作必须保持高语义密度，避免低层操作洪流 |
| 每回合必须以一个终结性的 `Large-Action` 收束 | 强化“行动机会稀缺”的策略意义，同时允许回合内先做局部整理 | 一个回合可以包含若干 `Small-Action`，但只能有一个结束回合的 `Large-Action` |
| 回合内允许执行 `Small-Action` | 让 `Memory-Notebook` 编辑等局部行为能自然发生，而不必强行占掉整回合 | 需要区分“不会结束回合”的步骤与“会推进世界”的终结动作 |
| 系统一次性展示该回合全部可感知信息 | 便于把“观察输入”压成最小必要上下文 | Presenter 应输出稳定的 `Perception-Bundle`，而不是碎片化 prompt |
| Player 必须写出文本形式的决策过程 | 这是训练价值的一部分，不只是 UI 文案 | 回合输入不只是动作，还包括 `Reason-Trace` |
| 游戏用专用内建 LLM 验证每一步动作的 `Reason-Trace` 是否合理 | 目标不是判定“绝对正确”，而是约束 Player 不能依赖游戏外记忆或作弊式跳步，同时给出可重试的反馈 | 需要一个逐步 validation phase，校验每个动作是否 grounded 于当前可见信息与允许使用的记忆 |
| `Memory-Notebook` 是最重要的游戏道具 | 它是连续性的主要载体，也是训练语料的高价值产物 | 世界状态中必须有第一类 notebook 容器，而不是临时字符串备注 |
| 外部 `pmux` Player 的有效回合输入作为同步信号 | 这给多 Player 世界提供了最简单的 barrier | 首版可以让外部 Player 负责触发 world tick，其他 LLM Player 在 barrier 内同步产生活动 |

## 核心玩法骨架

### 1. Perception-Bundle

每个回合开始时，系统向 Player 展示一个完整的 `Perception-Bundle`。

它应包含当前回合内 **全部允许 Player 感知到的信息**，但不泄露超出视角边界的世界真相。

这一步的目标不是“让信息很多”，而是“让输入最小但足够支撑一次真实取舍”。

### 2. Small-Action + Large-Action

一个回合不再被理解为“只能提交一个动作”，而是一个有边界的步骤序列：

- 先看到完整的 `Perception-Bundle`
- 然后执行零到多个 `Small-Action`
- 最后以一个终结性的 `Large-Action` 收束回合

`Small-Action` 的典型例子包括：

- 编辑 `Memory-Notebook`
- 其他不直接结束回合的局部整理或试探行为

当前首版的重点，不是把 `Small-Action` 做得很多，而是先把这种“两层动作”结构立起来。

### 3. Large-Action

每个回合，Player 必须最终提交一个 `Large-Action` 来结束本回合。

`Large-Action` 的设计目标不是丰富按钮数量，而是让一个动作本身就携带明确机会成本，例如：

- 探索一个区域
- 与某个对象或人物深入互动
- 处理一项生存/整理事务
- 推进一个明确目标
- 规避某类风险

地点间移动、细粒度房间操作、逐步式 parser 交互，不应成为首版核心。

### 4. Reason-Trace

Player 在提交每一步动作时，都必须同时提交 `Reason-Trace`。

`Reason-Trace` 的作用不是文学修辞，而是把如下内容外化出来：

- 我现在依据了哪些当前可见信息
- 我依据了哪些 notebook 中已有记忆
- 我为什么认为这个动作比其他动作更值得
- 我承认自己不知道哪些事

这使得游戏天然产出一类高价值的过程样本，而不是只保留最终动作标签。

### 5. Validation Phase

系统在接受每一步动作前，都需要先验证 `Reason-Trace` 是否 grounded。

当前明确的验证目标是：

- 不允许把当前不可见的信息当作既知事实
- 不允许把 notebook 中不存在的历史细节当作稳定记忆
- 不允许只给结论，不给足以支撑该结论的最小理由链

当前方向已经收敛为：语义判定主要由专用 LLM validator 承担，并在不通过时给 Player 反馈，允许其修正后重试。

当前尚未决定的是失败次数、代价模型和 validator 输出协议，但“未通过校验的输入不应直接推动世界前进”这一点是合理的首版约束。

### 6. Synchronous Resolution

当外部 `pmux` Player 给出有效回合输入后，世界开始这一回合的统一推进：

1. 收下外部 Player 的有效意图
2. 驱动其他内建 LLM Player 产生活动意图
3. 汇总所有 Player/NPC 的本回合动作
4. 根据规则统一归约世界结果
5. 写回下一回合的可见变化

这意味着世界的主节奏不是“谁先打字谁先动”，而是“围绕回合 barrier 的同步结算”。

## 为什么 `Amnesia` 很适合这条路线

`Amnesia` 在这里不是单纯题材皮肤，而是一个非常强的设计收缩器。

它同时解决了几件事：

- 为“Player 不能依赖脑内长时记忆”提供世界内解释
- 为 `Memory-Notebook` 的核心地位提供世界内解释
- 把“最小必要上下文”从工程优化上升成玩法本体
- 把错误记忆、缺失记忆、过度自信、复盘修正都自然纳入训练目标

换句话说，这个主题不是额外装饰，而是玩法约束、训练目标与交互协议的共同支点。

## 这条路线对 TextAdv 的直接工程含义

### 1. 世界核心要从“地点移动 demo”升级为“回合推进器”

当前 `go/look-around` 型原型仍有价值，但它更适合作为局部世界操作的演示，不再适合继续充当最终核心循环。

真正的核心应逐步转向：

- day / slot 调度
- Perception-Bundle 构造
- 回合内步骤接受与 notebook 更新
- `Large-Action` 提交
- `Reason-Trace` 校验
- 多方意图汇总
- 同步结算与写回

### 2. `Memory-Notebook` 应成为第一类状态

Notebook 不应只是调试文本，也不应只是 UI 辅助。

它应被建模为 Player 状态的一部分，并且后续很可能适合落在可增量编辑的持久文本容器上。

### 3. 多 Player 不应先走实时化

首版最简单、也最符合训练目标的多 Player 方案，是同步回合制：

- 一个外部 `pmux` Player 作为显式驱动者
- 若干内建 LLM Player 作为同回合参与者
- 世界只在 barrier 达成时推进一步

这样可以用最小复杂度获得：竞争、协作、信息不对称、资源冲突与行为对比。

### 4. 训练数据与运行时产物应天然同构

这条路线最有价值的一点是：运行时本身就会产出高度可用的训练材料。

至少可以自然沉淀出以下片段：

- `Perception-Bundle`
- `Reason-Trace`
- `Large-Action`
- validation 结果
- 世界反馈
- notebook 增量

因此，玩法协议本身就应尽量靠近未来的数据协议，而不是另起一套离线标注格式。

## 首版明确不追求的东西

- 不追求实时世界推进
- 不追求大地图和细粒度地点导航
- 不追求 parser 式自由输入覆盖一切动作
- 不追求首版就引入复杂的动作点经济或大量 `Small-Action` 品类
- 不追求把所有隐藏规则一次性做复杂
- 不追求先做“内容丰富的冒险游戏”，再事后嫁接训练目的

## 当前仍待定的问题

以下问题很重要，但目前不需要先钉死：

1. validator 的输出协议、反馈粒度与失败重试上限
2. `Small-Action` 是否要引入动作点数或其他预算约束
3. 哪些候选 `Small-Action` 只影响本地状态，哪些会触发世界级或多方级归约
4. 不同 Player 的 notebook 权限和可见性是否完全一致
5. 多方动作冲突时采用怎样的 resolution order
6. `Reason-Trace` 是否全部留作世界内可见材料，还是部分只用于系统验证与训练

## 推荐的下一步设计重心

在这份方向收敛成立之后，下一阶段最值得优先澄清的，不是剧情和地图，而是下面三件事：

1. `Perception-Bundle` 的最小结构
2. 回合步骤协议与 validator 输出协议
3. `Small-Action` 的分类边界与预算模型

只要这三件事明确，TextAdv 的世界结构、命令接口和多 Player 同步机制就都会开始自然收束。
