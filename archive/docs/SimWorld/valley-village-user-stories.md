# 山谷小村 User Story 与 Test Case 草案

> 用途：把山谷小村从“有潜力的场景”推进为一组可直接反推引擎实现与验证条件的正式工作样例。

## 文档定位

本文档不再扩写世界观，而是把现有高价值情境压缩成统一格式的 user story / test case。

它服务三个目的：

- 验证当前 world model 是否足够表达关键行为
- 为后续实现提供最小可验证目标
- 为未来的自动回放、评测与 post-training 数据生成提供模板

## 使用方式

每个 case 都按同一结构编写：

- User Story：这个场景想验证什么能力
- 初始世界状态：地点、主体、资源、任务、时间窗口
- 动作序列：谁在何时尝试做什么
- 预期世界事件：Ground Truth 层应发生什么
- 预期观察分发：谁应收到什么样的 observation
- 通过条件：实现必须满足什么
- 非目标：当前 case 不要求解决什么

## 共享前提

### 1. 共享地点图

默认使用 [valley-village.md](valley-village.md) 中定义的山谷小村地点图：

- 山口哨点
- 村口主路
- 中央空地
- 住屋区
- 水井
- 谷仓
- 工坊
- 农田
- 林地入口
- 后坡小路

### 2. 默认主体画像

除非单独说明，首批 case 默认使用 [inhabitant-profiles.md](inhabitant-profiles.md) 中的 `Human-Lite Villager`。

这意味着：

- 理解承诺、借还、插队、求助、通报
- 有粗粒度体力/休息压力
- 有公开失败与关系评价压力
- 暂不依赖高保真生理与情绪系统

### 3. 时间记法

本文统一用相对时间：

- `t0`：case 起点
- `t+5`：从起点起第 5 个时间单位

时间单位当前只要求可排序，不要求先绑定到“分钟”或“秒”。

### 4. 事件队列稳定性

首批 case 默认要求事件队列在以下条件下具有稳定顺序：

- 若两个 `Action Reservation` 的 `ScheduledTime` 相同
- 系统必须仍能按稳定规则决定先后

这里不强行规定 tie-break 的具体实现，但要求其结果可重放、可解释。

### 5. 真相与观察分离

所有 case 都默认遵守：

- `World Event` 记录世界真相
- `Observation Delivery` 记录某个主体何时、以何种 certainty 收到什么

实现不得用“所有主体立即知道全部真相”来偷过测试。

## VV-US-001 单工具冲突

### User Story

作为需要依赖稀缺工具完成任务的村中主体，
我希望世界能稳定处理“多人争抢同一关键工具”的情况，
这样成功与失败才会变成真实后果，而不是被文本糊过去。

### 初始世界状态

- 时间：`t0`
- 地点：木匠与阿苗都在工坊
- 资源：工坊内只有一把状态为 `usable` 的斧头
- 任务：
  - 木匠需要斧头去处理木料
  - 阿苗需要斧头去砍柴，完成时限更晚

### 动作序列

1. `t0`，木匠提交 `Take(斧头)` reservation
2. `t0`，阿苗也提交 `Take(斧头)` reservation
3. 队列按稳定顺序依次出队

### 预期世界事件

- 仅有一条 `transfer` 或 `take-success` 事件成功发生
- 第二条 reservation 在出队时因前置条件失效而生成 `failure` 事件
- 斧头最终只能有一个持有者

### 预期观察分发

- 木匠收到直接成功 observation
- 阿苗收到直接失败 observation，原因至少应包含“目标已不再可取得”
- 若同地点还有旁观者，可收到直接目击 observation，但不是必需项

### 通过条件

- 不允许出现同一把斧头被两个主体同时成功取得
- 第二个动作的失败必须由真实前置条件失效触发，而不是被静默吞掉
- 相同 `ScheduledTime` 下的结果必须可重放

### 非目标

- 不要求处理礼让、协商、争吵等更高层社交行为
- 不要求工具预约系统超出“先成功、后失败”的最小语义

## VV-US-002 路径回避

### User Story

作为不想在公开路径上暴露行踪的主体，
我希望能选择较慢但更隐蔽的路径，
这样“绕行”和“回避被看见”才在地点图层面成立。

### 初始世界状态

- 时间：`t0`
- 阿苗位于林地入口，目标是回到住屋区
- 石守将在 `t0` 到 `t+8` 之间沿中央空地附近巡查
- 可选路径：
  - 公开路径：林地入口 -> 农田 -> 中央空地 -> 住屋区
  - 隐蔽路径：林地入口 -> 后坡小路 -> 住屋区

### 动作序列

1. 阿苗选择 `MoveTo(住屋区)`，路径偏好为隐蔽
2. 石守按既定巡查路线移动
3. 系统根据边的 `VisibilityClass` 生成或不生成 sighting

### 预期世界事件

- 阿苗沿后坡小路完成移动，耗时高于公开路径
- 若石守未进入可直接观察的地点/边范围，则不生成 direct sighting

### 预期观察分发

- 阿苗收到自己的移动结果
- 石守不应因为“系统知道阿苗移动过”而自动收到真相
- 若有其他位于回路附近的主体，可按边语义收到 observation

### 通过条件

- 地点图本身就能表达“更慢但更隐蔽”的路径差异
- 不需要地点内格子，也能成立“绕开中央空地”的行为
- 可见性差异必须影响 observation，而不只是文本 flavor

### 非目标

- 不要求完整潜行系统
- 不要求处理朝向、蹲伏、房间内部遮挡

## VV-US-003 送达错位

### User Story

作为负责送达物品的主体，
我希望当接收者在我抵达前离开原地点时，
系统给出“目标不在场”的真实失败结果，
这样“错过”才会成为一类可验证行为。

### 初始世界状态

- 时间：`t0`
- 小递位于工坊，持有已修好的锄头
- 阿苗初始位于工坊，但已接受 `MoveTo(农田)` 的任务
- 小递的任务是把锄头交给阿苗

### 动作序列

1. `t0`，阿苗提交从工坊前往农田的移动 reservation
2. `t+1`，小递提交 `Deliver(锄头 -> 阿苗@工坊)` reservation
3. 阿苗先离开工坊
4. 小递的 deliver 在出队时重验“接收者是否在场”

### 预期世界事件

- 阿苗成功离开工坊并抵达农田
- 小递的 deliver 生成 `failure` 事件，原因是接收者不在预期地点
- 锄头不会被瞬移交付；它仍在小递手中或按失败模板回到原持有状态

### 预期观察分发

- 小递收到直接失败 observation
- 阿苗不应自动收到“已收到了工具”的错误 observation
- 若后续有人报告阿苗的新位置，这应作为新的 observation delivery，而不是追写旧事件

### 通过条件

- 送达动作必须依赖接收者在场这一前置条件
- 失败后世界状态一致，不得出现“交付失败但物品丢失”
- 系统为后续重规划留出口，但不要求自动生成补救动作

### 非目标

- 不要求自动寻找最近路径继续追送
- 不要求复杂的人际沟通策略

## VV-US-004 山口来客

### User Story

作为一个有对外瓶颈的聚落，
我希望陌生来客进入时，不同主体会按地点、职责和消息传递路径在不同时间获知，
这样“观察传播”才不是一句空话。

### 初始世界状态

- 时间：`t0`
- 陌生旅人出现在山口哨点
- 石守位于山口哨点
- 谷长位于中央空地
- 木匠位于工坊
- 阿仓位于谷仓

### 动作序列

1. `t0`，石守对旅人形成 direct sighting
2. `t+1`，石守提交 `Report(谷长, 山口有来客)`
3. `t+3`，谷长收到报告
4. `t+4` 以后，谷长可再决定通知木匠或阿仓，或暂不通知

### 预期世界事件

- 旅人的出现本身形成一条 `sighting` 或 `arrival` 事件
- 石守的报告形成独立 `report` 事件
- 不同接收者的获知时间不同

### 预期观察分发

- 石守在 `t0` 立即收到 direct + certain observation
- 谷长在报告送达后收到 reported observation
- 木匠与阿仓在未被通知前，不应自动得知来客存在

### 通过条件

- 世界不允许“一个公共事件自动瞬时广播给全村”
- 同一真相事件可以生成多条不同时间、不同 delivery mode 的 observation
- 后续反应分化是可能的，而不是被系统提前统一

### 非目标

- 不要求实现完整外交、交易、安检或敌意判定系统
- 不要求立刻给来客生成复杂背景故事

## VV-US-005 夜间传闻

### User Story

作为会依赖二手消息行动的村中主体，
我希望系统能区分“世界真相”“部分目击”和“转述传闻”，
这样调查、误判与谣言扩散才有真实基础。

### 初始世界状态

- 时间：夜间 `t0`
- 一名主体沿后坡小路移动
- 叶姨位于住屋区边缘，只能模糊看到一个身影朝谷仓方向附近移动
- 谷仓此时并没有发生失窃

### 动作序列

1. `t0`，真实移动事件发生
2. `t0`，叶姨获得 partial certainty 的 sighting observation
3. `t+2`，叶姨向阿仓转述“我好像看见有人夜里往谷仓那边去了”
4. `t+4`，阿仓可能据此发起 inspect 或 report 行动

### 预期世界事件

- Ground Truth 只记录“某主体夜间移动经过某路径”
- 不生成“谷仓失窃”这样的虚假真相事件
- 叶姨的转述形成 `report` 或 `rumor-spread` 事件

### 预期观察分发

- 叶姨收到 direct + partial observation
- 阿仓收到 reported + rumor observation
- 若后续检查谷仓无异常，应生成新的核查结果 observation，而不是 retroactively 改写 rumor

### 通过条件

- 传闻必须以 observation delivery 的 certainty 区分，而不是篡改 world event
- 后续核查能够证伪或削弱传闻，但不能抹掉其曾传播过的事实
- 即便没有地点内潜行几何，也能表达“夜里看错/看不清”的信息差

### 非目标

- 不要求复杂谣言心理学模型
- 不要求每个主体都有完整 belief graph

## VV-US-006 临时补位

### User Story

作为一个依赖有限劳力运转的小社会，
我希望当关键执行者在任务开始前临时失去可用性时，
系统能把失败结果回注给相关主体，并允许再分配，
这样“补位”才不是脚本台词。

### 初始世界状态

- 时间：`t0`
- 工坊需要在 `t+10` 前获得一桶水
- 小递原本负责去水井取水并送往工坊
- 叶姨在住屋区
- 谷长在中央空地

### 动作序列

1. `t0`，小递接受 `FetchWater -> DeliverTo(工坊)` 任务
2. `t+2`，小递因突发不适进入不可执行状态
3. 尚未出队的取水或送水 reservation 在出队时失败或被阻塞
4. 相关 observation 送达给谷长或叶姨
5. 谷长重新把任务分配给叶姨或其他主体

### 预期世界事件

- 小递状态变化形成 `incapacitated` 或等价事件
- 原任务链上至少有一条 reservation 失败或转 blocked
- 新的任务承诺可被创建并指向新的责任主体

### 预期观察分发

- 小递收到自身状态变化 observation
- 谷长收到“原执行者无法继续”的 observation
- 新接手者收到新的任务分配 observation

### 通过条件

- 任务失败不能只停留在内部状态；必须对相关主体形成可见后果
- 再分配必须建立在新的任务承诺或 reservation 上，而不是隐式瞬移完成
- 工坊最终要么收到延迟交付，要么收到明确的“尚未送达”状态

### 非目标

- 不要求自动求最优调度
- 不要求完整医疗或护理系统

## 由这些 case 反推出来的硬要求

在当前抽象下，这 6 个 case 都可以落在现有最小世界模型里，但它们也反推出几条不能省略的工程要求：

### 1. 队列顺序必须稳定

至少 `VV-US-001` 需要一个可重放的 tie-break 规则。

### 2. Observation Delivery 不能只是附属日志

至少 `VV-US-004` 和 `VV-US-005` 说明，observation delivery 本身就是一等对象。

### 3. 失败模板必须显式存在

至少 `VV-US-001`、`VV-US-003`、`VV-US-006` 都要求动作失败后能产出结构化结果，而不是只返回一段自然语言。

### 4. 任务与 reservation 不能混为一谈

至少 `VV-US-006` 说明：

- Task Commitment 是社会层承诺
- Action Reservation 是即将执行的未来动作

它们需要分别建模。

## 后续扩展方向

当这 6 个 case 全部跑通后，下一批更自然的扩展是：

- profile 切换 smoke test：把其中 1 到 2 个 case 改写成 `Household-Robot-Lite`
- 引入容器与库存层级
- 引入地点内语义锚点
- 引入更明确的 authority / permission 模型

在那之前，不建议把测试重心拉向：

- 精细战斗
- 连续物理
- 高保真情绪演算
